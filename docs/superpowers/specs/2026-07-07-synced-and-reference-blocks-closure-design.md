# 同步块 / 引用块收口设计稿

- 日期：2026-07-07
- 状态：待评审
- 适用项目：知栖桌面端
- 主题：阶段 3 - 同步块 / 引用块收口

## 1. 背景

知栖已经完成了同步块 / 引用块的第一版底座：

- 可以从连续选区创建同步块
- 可以插入同步实例和引用实例
- 引用块支持只读展示、前往原位置、取消同步
- 搜索已经能区分同步块内容和引用块内容
- 同步组已经具备 `groupId / instanceId / primaryInstanceId` 结构

但当前更像“功能已经出来”，还不是“可以长期放心使用”的状态。

问题不再是缺入口，而是有几条关键链路还没有完全闭合：

1. 有些底层数据规则还没有被正式钉死
2. 有些复杂块的行为规则还不够统一
3. 搜索、复制、导入导出、资源清理与同步组的联动还不够完整
4. 用户侧已经能用，但仍然会在少数场景里感到别扭或拿不准

因此，这一轮的目标不是继续横向扩功能，而是把同步块 / 引用块收成一个稳定、清晰、可长期演进的能力。

## 2. 收口目标

本轮收口按三层标准推进，但顺序明确为：

1. **B：数据层收口**
2. **A：用户层收口**
3. **C：产品层收口**

原因很直接：

- 没有数据层收口，越用越危险
- 只有数据层没有用户层，能力会显得笨重
- 产品层联动必须建立在前两层稳定之后

### 2.1 B：数据层收口

目标是：

- 不管怎么复制、删除、导入导出、备份恢复，都不产生脏数据
- 同步组、实例、主实例、资源引用之间的关系稳定可推导
- 搜索索引和页面内容不会出现幽灵结果或失效引用

### 2.2 A：用户层收口

目标是：

- 用户能一眼看懂当前块是什么
- 创建、查看、删除、取消同步、前往原位置的交互保持一致
- 同步实例、引用实例、缺失实例、复杂块回跳策略都有明确体验规则

### 2.3 C：产品层收口

目标是：

- 同步块 / 引用块不只是编辑器功能，而是正式进入搜索与知识关系体系
- 用户能从搜索结果和复用关系中理解“这段内容在哪些地方被使用”

## 3. 本轮不做

本轮明确不做以下内容：

- 同步块历史版本
- 同步块权限系统
- 同步块评论 / 协作态
- 同步块图谱视图
- 引用块悬浮预览卡片
- 嵌套同步块
- 白板 / 数据表 / 导图在副本里的原地深编辑
- 全新的“同步组管理中心”面板

这些都可能有价值，但不是这轮收口必须项。

## 4. 当前状态总结

### 4.1 已经具备的能力

- `synced_block` 已经作为页面中的实例容器块存在
- 工作区已经有 `syncedBlockGroups` 数据结构
- 主实例丢失时已经有迁移逻辑
- 最后一个实例删除后已经会清理空组
- 取消同步已经能把共享内容落回本地独立块
- 引用块已经具备基本只读样式和删除能力

### 4.2 当前最大的风险点

当前最需要优先处理的风险是：

**页面资源引用扫描还只看 `pages`，没有把 `syncedBlockGroups` 里的复杂块资源一起算进去。**

这意味着如果白板、数据表、导图等复杂块存在于同步组内容里，后续页面删除、资源过滤或清理孤儿资源时，存在误删真实仍在使用资源的风险。

这一点是本轮的 P0。

## 5. 收口决策

## 5.1 数据层决策

### 5.1.1 同步组中的复杂块资源算“真实引用”

凡是出现在 `syncedBlockGroups[].blocks` 中的白板、数据表、导图、媒体等资源，都必须被视为真实引用资源。

因此所有资源引用扫描都要同时覆盖两层：

- 页面 `pages[].blocks`
- 同步组 `syncedBlockGroups[].blocks`

不能只扫描页面表面块。

### 5.1.2 页面复制保持“继续指向原同步组”

本轮正式确认：

**复制页面时，如果页面中包含同步块 / 引用块，复制出来的新页面实例继续指向原同步组，只重写新的 `blockId` 和 `instanceId`。**

不自动断开，不自动派生新同步组。

原因：

- 这与当前实现一致，改动最小
- 复制页面本质上是复制一个“内容视图”，不是复制一份新的共享源
- 如果用户想脱离共享关系，复制后可以手动执行“取消同步”

这条规则必须写进测试，避免以后改着改着变成别的语义。

### 5.1.3 页面包导入导出保留包内共享关系

页面包导出时：

- 只导出被页面树实际引用到的同步组
- 同步组内容随包一起带走

页面包导入时：

- 重写 `groupId / instanceId / blockId`
- 保留包内实例之间原有的共享关系
- 不允许出现“导入了实例，但缺组内容”的残缺状态

### 5.1.4 JSON 备份 / 恢复按工作区全量保真

JSON 备份 / 恢复不做降级处理。

只要工作区里有同步组：

- `syncedBlockGroups`
- 组内容里的关系 rich text
- 主实例信息

都必须完整往返。

### 5.1.5 删除规则

以下规则在本轮锁死：

- 删除普通实例：只删该实例，组保留
- 删除主实例：重新选一个主实例
- 删除最后一个实例：组一并清理
- 取消同步：移除该实例，并在原位置落成本地独立块
- 删除页面：页面里所有同步实例一并移除，并触发组重算

### 5.1.6 非法数据兜底

除了创建入口外，下列路径也必须具备同步组数据修正或拒绝逻辑：

- 启动加载快照
- JSON 恢复
- 页面包导入

最少要处理这些坏数据：

- 组为空
- `primaryInstanceId` 指向不存在的实例
- 实例指向不存在的 `groupId`
- 同步组内容里嵌套 `synced_block`

## 5.2 用户层决策

### 5.2.1 四种状态统一

本轮将同步块 / 引用块 UI 明确分成四种状态：

1. 同步主实例
2. 同步普通实例
3. 引用实例
4. 缺失实例

它们的视觉识别必须稳定，不能靠用户猜。

### 5.2.2 编辑能力矩阵

本轮统一如下：

- **同步实例**
  - 文本类块：允许原地编辑并同步
  - 复杂块：显示内容壳，但编辑动作默认引导到主实例
- **引用实例**
  - 全部只读
  - 可预览
  - 复杂编辑统一走“前往原位置”

这里的复杂块包括：

- 代码块
- 普通表格
- 白板
- 数据表
- 导图

媒体块维持可预览展示，但不额外扩出复杂编辑规则。

### 5.2.3 手柄菜单规则统一

手柄菜单需要统一以下规则：

- 何时显示“前往原位置”
- 何时显示“取消同步”
- 何时只允许删除当前实例

建议规则：

- 引用实例：总是显示“前往原位置”和“取消同步”
- 同步普通实例：显示“前往原位置”和“取消同步”
- 同步主实例：不显示“前往原位置”，保留“取消同步”
- 缺失实例：只保留“取消同步”

### 5.2.4 焦点与键盘规则

本轮将把以下行为统一到测试里：

- 点击引用块时能获得明确焦点
- `Backspace / Delete` 可以删除引用实例
- 取消同步后焦点落到还原出的本地块
- 从“前往原位置”返回后，用户不会丢失阅读上下文
- 菜单锚点、焦点高亮、滚动跟随保持一致

## 5.3 产品层决策

### 5.3.1 搜索按实例命中

搜索命中的是哪个实例，就跳哪个实例，而不是一律跳主实例。

原因：

- 搜索结果反映的是用户看到内容的位置
- 实例页面本身就是用户当前知识上下文的一部分

如果后续需要查看主实例，作为二级动作提供“前往原位置”即可。

### 5.3.2 搜索来源标签保留区分

搜索结果继续区分：

- 同步块内容
- 引用块内容

不合并成单一来源。

### 5.3.3 复用关系可见化作为本轮最小产品联动

本轮产品层不强做完整图谱，但至少要保留一条最小联动：

- 主实例上的复用次数角标持续可见
- 后续可以从这个角标扩展出“查看使用位置列表”

本轮先确保计数和基础语义稳定，不强制同时上线完整去向面板。

### 5.3.4 页面关系归属暂不扩到 backlinks / mentions

同步组内部 rich text 已经可以携带页面关系，但本轮不把它直接并入页面底部的 backlinks / mentions 区。

原因：

- 这里会牵涉“关系归主实例页面还是实例页面”的产品定义
- 这会明显扩大阶段 3 的范围

本轮先把搜索语义收口，把 deeper relation ownership 留到后续页面关系化迭代。

## 6. 验收标准

### 6.1 B：数据层验收

- 同步组中的复杂块资源不会被误清理
- 删除实例、删除主实例、删除最后实例，组状态都正确
- 复制页面后实例继续连向原同步组，且测试覆盖
- 页面包导入导出后，共享关系保持正确
- JSON 备份恢复后，共享关系保持正确
- 启动加载时坏数据不会直接污染工作区

### 6.2 A：用户层验收

- 四种状态视觉和行为一致
- 文本类同步编辑顺手
- 复杂块回跳主实例规则一致
- 引用块删除、取消同步、前往原位置、聚焦反馈都稳定

### 6.3 C：产品层验收

- 搜索结果明确区分同步块内容与引用块内容
- 搜索跳转按实例落位
- 主实例复用计数逻辑稳定

## 7. 实施顺序

### 第一步：P0 / P1 数据层闭环

- 修正资源引用扫描
- 锁定复制页面语义
- 补导入导出 / 备份恢复 / 删除迁移测试
- 补坏数据兜底

### 第二步：P1 用户层统一

- 统一四种状态样式
- 统一复杂块编辑策略
- 补焦点与键盘回归
- 统一手柄菜单动作

### 第三步：P2 产品层最小联动

- 锁定搜索跳转语义
- 保持来源标签区分
- 稳定复用计数

## 8. 结论

这一轮的重点不是再给同步块 / 引用块“加更多能力”，而是把它们从“已经能用”收成“可以长期承载真实知识数据”的状态。

真正的完成标准不是某一个按钮出现了，而是这三层成立：

- 数据不坏
- 体验不别扭
- 搜索和知识系统里说得通

本轮按 **B -> A -> C** 推进，是最稳、最省返工的路径。
