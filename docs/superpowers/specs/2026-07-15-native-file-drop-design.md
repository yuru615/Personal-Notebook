# 原生左右分区文件拖放设计

## 目标

修复 Windows 安装版拖入文件时由网页拖放链路触发的 `zhiqi.exe` 栈溢出，并把拖放反馈从全局“导入收件箱”改为与实际落点一致的左右分区提示。

## 已确认问题

- `AppShell` 在任意子区域的 `dragenter` 上都显示全局收件箱遮罩，局部页面仅在 `dragover` 时阻止事件，因此提示与实际目标不一致。
- Windows 事件日志记录安装版 `zhiqi.exe 1.2.1` 在拖放时以 `0xc00000fd`（栈溢出）退出。
- 当前桌面路径会将整个拖入文件转为 `File`/`Uint8Array`，再通过 Tauri 命令传入 Rust；项目已具备按本机文件路径导入资产的原生命令，应使用该通道保存原始附件。

## 方案

桌面端启用 Tauri 原生文件拖放事件，获得文件真实路径与落点位置。应用根据落点是否位于侧栏宽度内，将一次拖放路由为 `sidebar` 或 `content`：

- 左侧侧栏：DOCX、PDF、Markdown、TXT 创建顶级页面；图片、音频、视频、压缩包及其它附件进入收件箱。
- 右侧正文：DOCX、PDF、Markdown、TXT 创建当前页面的子页面；图片、音频、视频、压缩包及其它附件追加到当前页面末尾。

拖放期间显示两个视觉分区，只高亮当前落点：左侧说明顶级页面/收件箱规则，右侧说明子页面/正文附件规则。取消全局收件箱的隐式兜底路径。

桌面端对原始文件使用路径导入资产，避免将整份原始文件经 WebView IPC 传递。DOCX/PDF 的文本解析仍在前端读取文件内容后完成；Word 内嵌图片沿用既有解析结果写入逻辑。

## 非目标

- 不改变页面树拖拽排序。
- 不修改浏览器版的文件选择、导入导出或资产格式。
- 不改变 DOCX/PDF 的标题、正文和附件保存规则。

## 验证

- 单元测试覆盖原生拖放的左右分区判断、进入/离开/投放状态，以及路径导入路由。
- 应用测试覆盖左侧创建顶级页、右侧创建子页、附件分别进入收件箱或当前正文。
- 完成前运行相关测试、全量前端测试、静态检查、前端构建和 Windows 安装包构建；再用实际安装版复测 DOCX 投放。
