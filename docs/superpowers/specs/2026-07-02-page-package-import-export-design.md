# Current Page Package Import and Export Design

## Goal

Replace the current full-workspace backup and restore entry points with page-package import and export.
The page menu should export the current page tree, not the whole workspace. Import should add a new top-level page tree, matching the placement of a newly created sidebar page.

The full-workspace backup and restore feature is intentionally removed from the user-facing app.

## User-Facing Behavior

- The page menu copy changes from full backup language to page package language, such as `导出当前页面` and `导入页面包`.
- Export packages the current page and all descendant pages.
- Import creates a new top-level page tree. The imported root page has `parentId: null`, like a sidebar-created page.
- Import does not overwrite, delete, or merge with existing pages.
- After import succeeds, the app navigates to the imported root page.
- Import confirms that a page package will be added before changing local data.

## Package Contents

The page package is a ZIP archive. It contains:

- A JSON manifest with package version, root page id, pages, referenced boards, data tables, mindmaps, and asset metadata.
- Asset files used by exported page content and referenced structured resources.

The package contains only resources reachable from the exported page tree. Unrelated workspace pages and resources are excluded.

## Identity and References

Import always creates copies.

- Page ids are regenerated.
- Block ids are regenerated where needed to avoid collisions.
- Board, data table, and mindmap ids are regenerated.
- Media asset ids are regenerated by writing imported files through the existing asset storage path.
- Page parent links and block/resource references are rewritten to point at the new ids.
- The imported root page becomes a top-level page.

This keeps imported packages isolated from existing workspace data.

## Architecture

Use the existing Tauri storage boundary and file-access pattern.

- Frontend owns menu actions, confirmation, progress display, final navigation, and store refresh.
- Rust storage owns package file reading and writing, ZIP processing, resource traversal, asset streaming, and SQLite writes.
- The frontend should pass the current page id to export and should receive the imported root page id from import.
- The current full-workspace archive commands should be removed from the UI path and replaced by page-package commands.

The implementation should avoid loading large package files into the WebView in desktop runtime. Desktop export/import should pass file paths to Rust, as the current workspace archive path commands do.

## Data Flow

Export:

1. User chooses `导出当前页面`.
2. Frontend asks for a ZIP save path with a default name based on the current page title.
3. Frontend calls a Tauri page-package export command with `pageId` and `path`.
4. Rust loads the current page tree, finds referenced boards, data tables, mindmaps, and assets, writes the ZIP, and emits progress.

Import:

1. User chooses `导入页面包`.
2. Frontend asks for a ZIP file path.
3. Frontend confirms that the package will be imported as a new top-level page.
4. Frontend flushes pending saves.
5. Frontend calls a Tauri page-package import command with `path`.
6. Rust reads the package, regenerates ids, writes assets and records, inserts the root page as a top-level page, and returns the new root page id.
7. Frontend refreshes the workspace and navigates to the imported root page.

## Compatibility

Existing full-workspace backup ZIP files are not page packages and should be rejected by the page-package import path with a clear failure message.

The new package manifest should include a type/version marker so future imports can distinguish page packages from old workspace archives.

## Error Handling

- Cancelled file dialogs should make no changes.
- Invalid ZIP files, missing manifests, unsupported versions, and workspace archive files should fail without modifying existing data.
- Import should write data in a transaction after validation whenever possible.
- Asset import failures should fail the import instead of leaving a partially connected page tree.
- User-facing copy should mention page-package import/export, not workspace backup/restore.

## Testing

Add focused coverage for:

- Page menu shows page-package copy and no full-workspace backup copy.
- Export passes the current page id and selected path to the page-package export command.
- Export includes the current page descendants and excludes unrelated pages.
- Import inserts a new top-level page tree and does not overwrite existing pages.
- Import regenerates ids and rewrites page/resource references.
- Import rejects full-workspace backup ZIPs or invalid package manifests.
- Rust archive tests verify package round trip with assets and structured resources.
