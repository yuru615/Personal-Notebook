# 白板功能接入设计

- 日期：2026-06-17
- 项目：`E:\Workspace\P001_Notion个人知识库\notion-web`
- 目标：在现有知识库中接入独立存储的白板能力，并通过命令菜单插入白板入口块，点击后进入白板编辑界面。

## 1. 范围结论

本次采用以下主方案：

- 白板数据独立存储
- 页面正文中的白板块只保存 `boardId`
- 命令菜单中新增“白板”
- 插入后在正文中显示白板缩略卡片
- 点击卡片进入独立白板编辑页

不采用以下方案：

- 不把整份白板 JSON 塞进普通块
- 不把白板直接做成页面树中的普通页面类型
- 不把旧白板整页以 iframe 或原始 HTML 形式硬挂进当前应用

## 2. 产品目标

用户可以在文档页内通过命令菜单插入一个白板入口块。该块显示白板缩略地图、标题和更新时间。点击后进入白板编辑页，完成编辑后仍然保存在当前知识库的本地数据中，并可再次从原文档页打开。

本次设计优先保证以下闭环：

1. 插入白板
2. 进入白板
3. 编辑白板
4. 返回文档
5. 下次重新打开仍能找到原内容

## 3. 非目标

以下能力不在本次实现范围内：

- 评论
- 提醒
- 协作
- 历史版本
- 白板内嵌页面引用
- 白板与数据库联动
- 图谱视图
- 白板块内直接原地编辑完整白板

## 4. 数据模型

### 4.1 页面块扩展

在现有 `BlockType` 中新增：

- `whiteboard`

新增块结构：

```ts
interface WhiteboardBlock extends BlockBase {
  type: 'whiteboard'
  boardId: string
}
```

将其纳入 `BlockRecord` 联合类型。

### 4.2 白板文档结构

新增独立的白板记录：

```ts
interface BoardRecord {
  id: string
  title: string
  snapshot: unknown
  createdAt: string
  updatedAt: string
}
```

说明：

- `snapshot` 第一版直接兼容旧 `Flowboard` 的白板数据结构
- 第一版不强行重构旧白板底层 schema
- `title` 独立存储，供白板页顶部和文档中的白板卡片共同使用

### 4.3 工作区快照扩展

现有 `WorkspaceSnapshot` 从：

```ts
{
  pages: PageRecord[]
  settings: WorkspaceSettings
}
```

扩展为：

```ts
{
  pages: PageRecord[]
  boards: BoardRecord[]
  settings: WorkspaceSettings
}
```

### 4.4 迁移规则

为兼容旧本地数据：

- 若历史快照中没有 `boards` 字段，则启动时自动补 `[]`
- 页面数据、设置数据保留原逻辑
- 这是一次无损升级，不要求用户手动迁移

## 5. 存储策略

### 5.1 存储原则

- 页面和白板分离存储在同一工作区快照中
- 白板块只保存 `boardId`
- 白板内容不复制到页面块里

### 5.2 块删除与复制规则

- 删除白板块：只删除块，不自动删除对应白板文档
- 一个白板允许被多个块引用
- 复制白板块：第一版复制引用，不复制底层白板内容

这样可以避免误删，并为后续“多个页面引用同一白板”保留能力。

## 6. 命令菜单与插入流程

### 6.1 命令菜单项

在现有 `/` 命令菜单中新增：

- 标题：`白板`
- 描述：`插入一个可点击进入的白板缩略入口`

### 6.2 插入行为

用户选择“白板”后，系统立即：

1. 创建空白 `BoardRecord`
2. 在当前位置插入 `whiteboard` 块
3. 将新建 `boardId` 绑定到该块

不增加确认弹窗。

## 7. 文档页中的白板块

### 7.1 展示结构

白板块采用卡片式结构，包含：

- 左侧：缩略地图预览
- 中间：白板标题、更新时间
- 右侧：进入提示

### 7.2 交互规则

- 点击整块：进入白板编辑页
- 悬停整块：轻微高亮
- 保留现有块手柄交互：拖动、删除、复制

### 7.3 标题来源

- 默认标题：`未命名白板`
- 白板页中支持重命名
- 文档页中的白板块直接读取 `BoardRecord.title`

不在块内额外冗余存一份标题，避免双写不一致。

## 8. 路由与页面结构

### 8.1 路由

新增白板路由：

- `/pages/:pageId/boards/:boardId`

其中：

- `pageId` 表示从哪个文档页进入
- `boardId` 表示打开哪个白板

### 8.2 页面壳

白板页继续使用当前知识库应用壳：

- 左侧保留知识库侧边栏
- 中间主区切换为白板编辑器
- 不拆成独立站点或单独应用

### 8.3 返回路径

从白板页点击返回后，回到来源文档页：

- `/pages/:pageId`

## 9. 白板编辑页设计

### 9.1 顶部操作

第一版保留以下操作：

- 返回文档页
- 白板标题编辑
- 导出
- 导入
- 清空
- 缩放比例显示
- 重置视图

### 9.2 页面布局

- 左侧：现有侧边栏
- 顶部：白板标题与操作栏
- 主区：白板画布
- 第一版不增加右侧属性面板

## 10. 旧白板能力迁移边界

### 10.1 本次保留

迁移旧 `Flowboard` 的核心能力：

- 画布平移与缩放
- 基础图形/线条/文本
- 选中、移动、删除
- 本地保存
- 导入导出
- 缩略图生成

### 10.2 本次不做

- 高级多选编组
- 评论
- 提醒
- 历史版本
- 白板内页面卡片
- 白板与页面双向引用

### 10.3 迁移方式

采用“迁移核心引擎，重做外壳”的方式：

- 尽量复用旧白板的数据、绘制、导入导出逻辑
- 不直接照搬旧 `index.html`
- 用 React 容器接入当前知识库的路由、标题栏和存储体系

## 11. 缩略图策略

第一版采用运行时缩略预览：

- 有内容时展示缩略图
- 空白白板展示占位态
- 第一版不做持久化图片缓存

这样可以避免额外的缓存失效与同步复杂度。

## 12. 导入导出约束

本次白板导入导出先独立于页面 Markdown 包：

- 白板页内提供白板自身导入导出
- 页面级 JSON 备份应包含 `boards`
- Markdown 页面包第一版不强求完整携带白板二进制或图片快照

这可以保证工作区备份完整，同时不扩大 Markdown 导出复杂度。

## 13. 错误处理

### 13.1 丢失白板引用

若块中的 `boardId` 找不到对应 `BoardRecord`：

- 白板块显示“白板不存在”异常态
- 点击不进入编辑页
- 后续可在块菜单中补“重新创建白板”或“替换白板”，但本次可先只做异常提示

### 13.2 非法路由

若访问的 `boardId` 不存在：

- 白板页显示空态或未找到提示
- 保留返回文档页入口

## 14. 测试范围

至少覆盖以下测试：

1. 历史快照无 `boards` 时可成功启动并补空数组
2. 选择命令菜单“白板”后会创建 `BoardRecord` 与 `whiteboard` 块
3. 点击白板块会跳转到正确的白板路由
4. 修改白板标题后，文档页白板块同步显示新标题
5. 删除白板块不会删除底层白板文档
6. 导出 JSON 时包含 `boards`
7. 丢失 `boardId` 时页面显示异常态而非崩溃

## 16. 实现建议

实现顺序建议如下：

1. 扩展类型与快照结构
2. 扩展仓库层与启动迁移
3. 扩展 store 的白板创建、读取、更新动作

## 17. 结论

本设计的重点不是把旧白板原型原样搬进来，而是把它收编为当前知识库中的一种独立内容对象。页面负责组织入口，白板负责承载自由画布内容，二者通过 `boardId` 关联。这一结构能在本次最小代价下完成闭环，并为后续白板复用、关系能力和统一备份留出空间。
