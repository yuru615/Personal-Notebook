# 文字 PDF 导入与统一文件拖放设计

**日期：** 2026-07-14
**状态：** 已获用户确认

## 目标

在既有 DOCX 拖放导入的基础上，支持本地文字型 PDF 转为可编辑页面，并统一不同文件拖入左侧页面树空白区与当前页面正文时的落点规则。

## 拖放规则

| 文件与目标 | 左侧页面树空白区 | 当前页面正文 |
| --- | --- | --- |
| DOCX、文字型 PDF、Markdown、TXT | 新建顶层页面 | 创建子页面与父页面 `child_page` 块 |
| 图片、音频、视频、ZIP、其他普通附件 | 收件箱 | 在当前页面末尾追加对应的媒体/文件块 |
| `.zhiqi` 页面包、完整工作区备份 | 保持明确的导入入口 | 保持明确的导入入口 |

一次拖入包含多种文件时，按文件逐类分流并按顺序写入，避免并发保存覆盖：文档先创建页面，其余附件再进入收件箱或当前页面末尾。

## PDF 范围

首版只支持带可选中文本层的 PDF，全程本地解析：

- 文件名作为页面标题；若 PDF 元数据存在标题则优先采用元数据标题。
- 根据文本位置和字号还原一级至三级标题及正文段落。
- 保留原始 PDF 作为新页面末尾附件。
- 不提取 PDF 内嵌图片，不还原复杂表格、多栏版式、批注、表单或扫描件 OCR。
- 没有可提取文字的扫描件会提示当前版本仅支持文字型 PDF，不创建半成品页面。

## 技术决策

使用 `pdfjs-dist` 的文字提取核心与 Worker，动态加载，只在导入 PDF 时进入运行时和构建产物。不会引入在线服务或上传文件；不使用完整 PDF 阅读器、示例资源、CMap 与字体包。依赖下载约 9.2 MB、安装目录约 35.8 MB；预计最小运行时构建产物增加约 0.7–1.5 MB。

## 验证

1. PDF 文本行分组与标题层级映射单元测试。
2. PDF 导入错误、空文本、元数据标题、原始附件测试。
3. 左侧与当前页对 DOCX/PDF/Markdown/TXT/附件混合拖放的分流测试。
4. 全量测试、lint、前端构建与 Windows Tauri 打包验证。
