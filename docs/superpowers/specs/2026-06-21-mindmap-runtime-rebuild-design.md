# 思维导图运行时重做设计

- 日期：2026-06-21
- 项目：`E:\Workspace\P001_Notion个人知识库\notion-web`
- 状态：已确认，待实现
- 目标：放弃当前脆弱的 `mindmap-v2` 交互实现，保留思维导图页面入口与数据模型，基于白板已经稳定的画布交互能力，重做一套独立的思维导图运行时。

## 1. 结论

本次采用下面这条路线：

1. 不再继续修补当前 `src/components/mindmap-v2/legacy/app.js`。
2. 保留思维导图页面入口、路由、基础数据结构。
3. 复制白板中已经稳定的画布交互能力，做一份独立的思维导图运行时。
4. 新运行时只服务思维导图，不与白板共享同一套运行时代码。
5. 当前思维导图内容允许清空，不做旧交互状态兼容。

一句话总结：

`复用白板成熟的画布底座，但把思维导图作为独立产品重做。`

## 2. 为什么当前版本判定作废

当前思维导图的核心问题不是单点 bug，而是交互模型已经失稳：

- 节点选中、双击编辑、拖拽、失焦、重渲染互相打架。
- 自动化验证可以通过，但人工点击路径不稳定。
- 修复一个点击问题，常常会带出另一个选择或拖拽问题。
- 现有 `legacy/app.js` 已经变成一个大体量、强耦合的运行时文件，不适合继续叠补丁。

继续修补的结果大概率是：

- 编辑态判断继续复杂化
- 事件补偿链继续增长
- 人工操作和自动验证继续不一致
- 后续多选、框选、吸附、排序预览会越来越难维护

所以这次不再把“修好当前版本”作为目标，而是明确切换到“保留产品目标，重做运行时”。

## 3. 方案比较

### 方案 A：继续修当前版本

优点：

- 眼前改动最小

缺点：

- 根问题不在单点逻辑，而在整体交互架构
- 后续每个新增能力都会继续叠在脆弱基础上
- 很难保证人工交互稳定

结论：

- 不采用

### 方案 B：直接共用白板运行时

优点：

- 复用比例最高

缺点：

- 白板是自由画布，思维导图是树结构，语义不同
- 思维导图会被迫带上很多白板不需要禁用的能力
- 白板后续优化容易反向影响思维导图

结论：

- 不采用

### 方案 C：复制白板底座，做独立思维导图运行时

优点：

- 直接站在白板现有稳定能力上
- 画布交互成熟，思维导图语义独立
- 白板和思维导图互不污染
- 后续可以专门为思维导图继续打磨

缺点：

- 前期有一笔复制和裁剪成本

结论：

- 采用

- 根节点与普通节点的视觉层级

## 5. 交互基线

本次重做统一采用下面这套交互规则，不再混用多套判断：

- 单击节点：只选中，不进入编辑
- 双击节点：进入编辑
- 选中节点后按 `Enter`：创建同级节点，并插入到当前节点下方
- 选中节点后按 `Tab`：创建子节点，并进入新节点编辑态
- `Delete` / `Backspace`：删除当前节点
- 鼠标滚轮：继承白板的缩放能力
- Shift + 点击：多选
- Shift + 拖拽：框选
- 点击空白：取消选中并退出编辑

这里的重点是：

- 选择和编辑彻底分离
- 编辑只依赖明确的双击路径
- 新节点创建后直接进入编辑态

## 6. 拖拽与吸附规则

拖拽节点时，要满足下面这些规则：

- 被拖动节点跟随鼠标移动
- 遇到可吸附父节点时，显示明确的目标反馈
- 不再默认丢到最后一个子节点位置
- 要显示“松手后会挂到哪个父节点、插入到哪个位置”的虚线预览
- 预览包含两部分：
  - 指向父节点的连接预览
  - 表示插入序号的落点预览

最终落位逻辑：

1. 先判断目标父节点是否合法
2. 再计算插入序号
3. 松手后按预览结果真正改树结构

## 7. 连线规则

本次重做的连线规则明确如下：

- 连线统一从节点左右中点发出
- 不根据悬停或临时状态漂移锚点
- 节点交换位置后，连线方向可以变化，但端点仍固定在块的左右中点
- 第一版就支持三种样式切换：
  - 直线
  - 曲线
  - 折线

曲线目标：

- 比当前版本更克制
- 在二级节点数量多时仍然可读
- 避免出现明显“没有接在块上”的视觉问题

## 8. 收起展开规则

- 悬停父节点时，显示独立的收起控制区域
- 控制区域与节点本体保持约 6px 间距
- 点击收起后，子树隐藏，并显示分支数量
- 右侧分支的收起标记显示在右侧
- 左侧分支的收起标记显示在左侧
- 收起数字和按钮都需要可点击，命中区域要比视觉图标略大

## 9. 视觉规则

### 9.1 页面外壳

- 思维导图页面外壳与白板一致
- 左上角返回页面
- 顶部工具栏悬浮在画布内部
- 进入思维导图后铺满，不再显示左侧知识库侧栏

### 9.2 节点视觉

- 根节点与普通节点必须有明显区分
- 不能只靠字体大小区分
- 第一版就加入颜色层级区分
- 选中态和悬停态严格区分，不允许悬停把选中样式改掉

### 9.3 多选与框选反馈

- 多选后的节点选中反馈必须明显
- 框选中的节点需要有成组的清晰视觉反馈
- 不允许出现“其实选中了，但看不出来”的状态

## 10. 历史记录规则

上一步 / 下一步 要继承白板成熟实现，但只记录节点操作：

- 创建节点
- 删除节点
- 重命名节点
- 拖拽改层级
- 拖拽改顺序
- 收起展开
- 样式切换

不纳入历史记录的内容：

- 视角移动
- 视角缩放
- 临时悬停态
- 预览态

## 11. 数据策略

- 允许清空当前思维导图内容后重新建立
- 不兼容当前运行时临时交互状态
- 尽量保留现有 `MindmapRecord` 主体字段，避免牵连知识库其他部分
- 新运行时内部可以重建自己的快照结构，但宿主侧仍以思维导图树数据为真相

这意味着：

- 宿主数据仍然是思维导图树
- 运行时只负责交互和渲染
- 不把思维导图降级成一组白板自由元素

## 12. 文件边界

建议保留 `mindmap-v2` 目录，但把它当成真正的重做版本，而不是旧版本补丁区。

计划涉及的主要区域：

- `src/components/mindmap-v2/`
  - React 外壳
  - 文档桥接
  - 数据模型转换
  - 运行时入口
- `src/components/mindmap-v2/legacy/`
  - 新的独立运行时 HTML / CSS / JS

边界原则：

- 白板代码不直接改造成思维导图代码
- 思维导图代码不反向抽象回白板
- 本次优先复制和裁剪，不做共享内核抽象

- 拖拽改父级

## 14. 验收标准

本次重做完成后，至少满足下面这些标准：

1. 人工双击现有节点，稳定进入编辑态。
2. 单击只选中，不误入编辑。
3. `Enter` 在当前节点下方插入同级节点。
4. `Tab` 创建子节点并进入编辑。
5. 拖拽时能明确看到将要挂接的父节点和插入位置。
6. Shift 多选和框选反馈清晰。
7. 缩放和平移体验与白板一致。
8. 上一步 / 下一步不再回退画布视角。
9. 根节点和普通节点视觉区别明显。
10. 白板功能不受这次重做影响。
