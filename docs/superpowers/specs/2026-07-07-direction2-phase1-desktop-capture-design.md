# Direction 2 Phase 1 桌面端捕获入口设计

- 日期：2026-07-07
- 状态：待评审
- 适用项目：知栖桌面端
- 主题：方向 2 第一阶段最小闭环

## 1. 背景

方向 1 解决的是“更强的组织与检索”，方向 2 要解决的是“更快地把内容带进来，并且全程本地、顺手”。

但方向 2 的能力面很大：全局快捷键、小窗速记、剪贴板捕获、拖拽导入、托盘入口、收件箱、本地文件联动、自动备份，都属于这个方向。现在如果一口气铺开，范围会很快失控。

所以这一阶段只做一个最小闭环：

1. 桌面端有一个稳定的系统入口
2. 工作区里有一个固定的 `收件箱`
3. 托盘可以直接创建普通笔记，或直接打开 `收件箱`

先把这条链路做顺，再往上叠全局快捷键、剪贴板捕获和快速输入窗。

## 2. 本轮目标

本轮目标只包含四件事：

- 在左侧页面结构中增加固定的系统分组
- 在系统分组中增加固定系统页 `收件箱`
- 调整托盘菜单为桌面端常用入口
- 让托盘动作和前端页面导航真正打通

验收标准不是“功能看起来像方向 2”，而是：

- `收件箱` 稳定存在
- 托盘 `新建笔记` 能立刻创建并打开普通页面
- 托盘 `打开收件箱` 能稳定跳到 `收件箱`
- 不引入新的一套捕获数据模型

## 3. 本轮不做

本轮明确不做：

- 全局快捷键速记
- 独立的小窗快速输入
- 剪贴板监听或自动捕获
- 拖拽文件自动入库
- 托盘里的最近页面列表
- 捕获内容“独立页面 / 每日归档”设置界面
- 自动备份、崩溃恢复、多窗口保护
- 本地文件原路径联动

这些都保留到方向 2 的后续阶段。

## 4. 方案选择

本轮比较过三种做法。

### 方案 A：系统分组 + 固定 `收件箱` + 托盘入口

这是本轮采用的方案。

做法：

- 把 `收件箱` 当成一张正常页面
- 通过工作区设置记录它的页面 id
- 在侧边栏单独渲染到系统分组
- 托盘通过事件通知前端“新建笔记”或“打开收件箱”

优点：

- 复用现有页面、编辑器、搜索、导入导出、持久化链路
- 不需要新建“捕获项”模型
- 范围小，风险低，适合 Phase 1

缺点：

- `收件箱` 现在还只是一个系统页，不是完整的捕获中心
- 后续如果加多种系统页，再补统一结构

### 方案 B：单独做一套 Inbox 数据模型

不采用。

做法是把 `收件箱` 做成独立实体，例如 `inbox_items`、`capture_entries` 之类，再额外开发展示、搜索、导出和整理流程。

问题是它会立刻带来第二套数据结构、第二套路由语义和第二套维护成本。对当前阶段来说太重。

### 方案 C：不做系统分组，直接把 `收件箱` 当普通顶层页

不采用。

优点是改动更少，但缺点也明显：

- 系统入口和普通页面混在一起
- 后续加托盘、快捷捕获时没有明确落点
- 用户很难感知这是“系统入口”，而不是一张普通笔记

## 5. 已确认的产品决策

以下决策已经固定：

- 左侧新增固定系统分组
- 系统分组里先只放一个固定系统页：`收件箱`
- 托盘菜单调整为：
  - `打开知栖`
  - `新建笔记`
  - `打开收件箱`
  - `退出`
- 托盘 `新建笔记` 创建的是普通页面，不进 `收件箱`
- 这个普通页面创建在 `我的页面` 顶层
- 创建后立即打开，直接进入编辑
- `收件箱` 主要服务未来的快速捕获内容
- 未来“独立页面 / 每日归档”的捕获存储方式会做成设置项
- 但本轮不提前暴露这个设置开关，避免出现死开关

## 6. 数据与架构设计

### 6.1 `收件箱` 仍然是一张普通页面

本轮不新增 page type，不新增独立表，不新增 capture item。

`收件箱` 直接复用普通 `PageRecord`，这样它天然拥有：

- 编辑器能力
- 页面属性能力
- 搜索能力
- 导入导出能力
- 本地持久化能力

### 6.2 用设置记录 `收件箱` 身份

本轮在 `WorkspaceSettings` 中新增一个轻量字段：

- `inboxPageId?: PageId`

含义：

- 这不是一张“特殊类型页面”
- 只是工作区设置里记住“哪一张普通页面当前承担收件箱角色”

这样做比改 `PageRecord` 更轻，也更符合当前只需要一个系统页的范围。

### 6.3 启动时自动修复 `收件箱`

bootstrap / snapshot 归一化时增加一个很小的修复规则：

- 如果 `settings.inboxPageId` 指向的页面存在：直接使用
- 如果字段缺失，或指向的页面已经不存在：自动创建一张顶层页面作为新的 `收件箱`
- 新建页面默认标题为 `收件箱`
- 新建后把它的 id 写回 `settings.inboxPageId`

这样即使用户删掉了旧 `收件箱`，系统也能在后续启动或打开收件箱时自愈。

### 6.4 左侧侧边栏增加系统分组

当前侧边栏已有：

- `星标置顶`
- `共享页面`
- `我的页面`

本轮新增：

- `系统`

顺序建议为：

1. `系统`
2. `星标置顶`
3. `共享页面`
4. `我的页面`

`系统` 分组里本轮只渲染 `收件箱`。

`收件箱` 不再重复显示在 `我的页面` 分组里，因此主页面树渲染时要排除 `settings.inboxPageId`。

### 6.5 托盘动作通过事件桥接前端

当前托盘已经有 Rust -> 前端事件模式，退出流程就是这样做的。

本轮沿用同一模式，新增两个桌面事件：

- `zhiqi://tray-new-note`
- `zhiqi://tray-open-inbox`

Rust 侧负责：

- 处理托盘菜单点击
- 显示主窗口并聚焦
- 向前端发事件

前端负责：

- 监听托盘事件
- 等待工作区 bootstrap 完成
- 执行页面创建或页面跳转

这样不需要在 Rust 侧重做页面创建逻辑，仍然让页面数据改动走现有 store / repository 链路。

## 7. 交互规则

### 7.1 托盘 `打开知栖`

- 显示主窗口
- 恢复并聚焦
- 不额外改动当前页面

### 7.2 托盘 `新建笔记`

- 先确保主窗口显示
- 前端等待 bootstrap 完成
- 调用现有 `createPage(undefined, { setCurrent: true })`
- 页面创建在 `我的页面` 顶层
- 创建后直接跳转到该页面

### 7.3 托盘 `打开收件箱`

- 先确保主窗口显示
- 前端等待 bootstrap 完成
- 读取 `settings.inboxPageId`
- 如果缺失或失效，先触发收件箱修复
- 跳转到 `收件箱`

### 7.4 `收件箱` 页面本轮保持普通编辑体验

为了保持范围最小，本轮不额外加这些限制：

- 不锁定标题编辑
- 不锁定图标和封面
- 不为 `收件箱` 新做专属页面头

也就是说，`收件箱` 目前只是“位置和入口特殊”，不是“编辑器行为特殊”。

如果后续需要把它做成真正的系统页，再补只读规则或系统页样式。

## 8. 错误处理与边界

### 8.1 `inboxPageId` 失效

如果 `settings.inboxPageId` 对应页面不存在：

- bootstrap 时修复
- 托盘 `打开收件箱` 时也要兜底修复

这样不依赖用户手动恢复。

### 8.2 托盘事件早于页面完成加载

如果托盘事件到达时工作区还没 bootstrap 完成：

- 前端事件处理先等待已有 bootstrap promise
- bootstrap 完成后再创建页面或导航

避免托盘动作打到空 store 状态。

### 8.3 `收件箱` 与页面树重复

系统分组和 `我的页面` 分组不能同时显示同一张 `收件箱` 页面。

因此：

- 系统分组单独渲染 `收件箱`
- 主页面树按 `inboxPageId` 过滤掉它

### 8.4 浏览器环境保持兼容

本轮托盘事件只在桌面端生效。

浏览器环境下：

- 不监听托盘事件
- `收件箱` 仍可作为普通页面存在
- 不引入额外报错

## 9. 测试要求

本轮至少补下面几类回归：

### 9.1 store / bootstrap

- 首次空工作区启动时会自动生成 `收件箱`
- `inboxPageId` 缺失时会自动修复
- `inboxPageId` 指向失效页面时会自动修复

### 9.2 sidebar

- 左侧出现 `系统` 分组
- `收件箱` 出现在 `系统` 分组中
- `收件箱` 不再重复出现在 `我的页面`
- 系统分组支持展开 / 折叠

### 9.3 tray / app routing

- 托盘 `新建笔记` 会创建顶层普通页面并跳转
- 托盘 `打开收件箱` 会跳转到 `收件箱`
- 托盘事件在 bootstrap 未完成时也能正常执行

### 9.4 Rust tray

- 新菜单 id 与动作映射正确
- 旧的显示窗口 / 退出行为不回归

## 10. 后续阶段接口

本轮完成后，方向 2 可以沿着下面顺序继续长：

1. 全局快捷键拉起快速输入
2. 快速输入默认进入 `收件箱`
3. 剪贴板捕获进入 `收件箱`
4. 再暴露“独立页面 / 每日归档”设置项
5. 再补托盘最近页面、本地文件联动、自动备份

也就是说，本轮做的是“入口地基”，不是完整的捕获系统。

## 11. 结论

这一版的核心判断是：

- `收件箱` 不需要先变成一套新模型
- 它先作为一张普通页面，挂到系统分组里就够用
- 托盘也不需要直接操作数据库，只要把动作桥接到现有前端 store 即可

这条路径最轻、最稳，也最适合现在马上进入实现。
