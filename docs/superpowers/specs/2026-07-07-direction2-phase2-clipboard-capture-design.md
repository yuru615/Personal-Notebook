# Direction 2 Phase 2 剪贴板捕获 v1 设计
- 日期：2026-07-07
- 状态：待评审
- 适用项目：知栖桌面端
- 主题：方向 2 第二阶段最小剪贴板捕获闭环

## 1. 背景

方向 2 要解决的是“更快地把内容带进来，而且全程本地、顺手”。  
第一阶段已经把桌面入口地基铺好：固定 `收件箱`、左侧 `系统` 分组、以及托盘入口已经打通。

接下来最自然的一步，不是立刻上全局快捷键或复杂捕获中心，而是先补一个足够轻的剪贴板捕获能力：

1. 用户在别的应用里复制内容
2. 知栖识别到“这是可收进来的内容”
3. 给出一个很轻的“保存到知栖”提示
4. 用户确认后，把内容放进 `收件箱`

这样能把“先收进来，后整理”的桌面端工作流跑通，同时又不把产品一下子做成重型后台监听器。

## 2. 本轮目标

本轮目标只包含以下几件事：

- 新增剪贴板捕获系统设置
- 支持桌面端检测新的可捕获剪贴板内容
- 用轻提示方式让用户决定是否保存
- 保存后统一落到 `收件箱`
- 只复用现有页面、块、资源和持久化能力，不新开第二套捕获模型

验收标准不是“已经像完整桌面收集箱”，而是：

- 用户可以明确关闭或开启这项能力
- 开启后，只在检测到新的可支持内容时出现提示
- 用户点击保存后，内容稳定进入 `收件箱`
- 不干扰知栖当前页面里的正常复制粘贴

## 3. 本轮不做

本轮明确不做：

- 手动捕获模式
- 剪贴板历史列表
- 后台静默自动入库
- 来源应用识别
- PDF / Word / 压缩包 / 任意文件附件块
- 收件箱专属卡片 UI
- 收件箱专属数据模型
- 多窗口同步捕获协调
- 全局快捷键拉起捕获

这里“手动捕获模式”不做，是因为它本质上没有额外价值。  
如果用户想把内容精确放到某个页面，本来就可以复制后直接到目标页面粘贴，不需要再额外包装成一套“手动捕获”系统。

## 4. 已确认的产品决策

以下决策已经固定：

- 剪贴板捕获设置项只有两个状态：
  - `关闭`
  - `复制后提示保存到收件箱`
- 默认值为 `关闭`
- 普通页面里的复制粘贴体验不改
- 只有“暂时来不及整理、先收进来”的内容，才通过半自动捕获进入 `收件箱`
- 捕获内容统一默认进入 `收件箱`
- 这轮只支持现有编辑器已经具备稳定承载能力的内容类型

## 5. 支持内容范围

### 5.1 本轮支持的内容

本轮支持以下五类：

1. 纯文本
2. 链接文本
3. 图片
4. 音频
5. 视频

### 5.2 本轮不支持的内容

以下内容即使出现在剪贴板里，也先忽略：

- PDF
- Word
- Excel
- 压缩包
- 任意未知文件
- 同时混合多种复杂剪贴板格式，但无法稳定判定主内容时

这样做的原因很简单：当前编辑器已经有成熟的文本块和媒体块，也已经有成熟的资源导入与资产持久化链路；但并没有一类现成、通用的“附件块”。  
如果这轮把任意文件一起纳入，就会把范围从“接现有地基”升级成“新增块类型”，不符合最小落地目标。

## 6. 内容如何落到收件箱

### 6.1 收件箱仍然是普通页面

本轮继续复用第一阶段的结论：

- `收件箱` 仍然是一张普通页面
- 不新建 `capture_item`
- 不新建 `clipboard_entry`
- 不新建单独的数据库表

这样可以直接复用：

- 普通页面编辑器
- 现有块渲染
- 现有搜索
- 现有导入导出
- 现有本地持久化

### 6.2 每次捕获写入的结构

每次成功保存时，往 `收件箱` 末尾追加一组普通块：

1. 一行文本说明块：`剪贴板捕获 · YYYY-MM-DD HH:mm`
2. 实际内容块
3. 一个空行分隔块

内容块映射规则：

- 纯文本 -> 段落块
- 链接文本 -> 段落块（保留链接文本本身，必要时可附带链接语义）
- 图片 -> 图片块
- 音频 -> 音频块
- 视频 -> 视频块

本轮不新做“捕获卡片”或“捕获容器块”。  
先用普通块把内容真正存下来，比额外做一层样式壳更重要。

## 7. 设置设计

本轮在系统设置里新增一个枚举设置，例如：

- `clipboardCaptureMode: 'off' | 'prompt_to_inbox'`

语义：

- `off`：关闭剪贴板捕获，不监听提示
- `prompt_to_inbox`：检测到新的可捕获内容时，提示用户是否保存到 `收件箱`

这里不保留 `manual` 模式，是为了避免重复设计已有编辑器行为。

## 8. 交互规则

### 8.1 关闭状态

当设置为 `关闭` 时：

- 不监听剪贴板变更
- 不出现提示
- 用户照常在任何页面复制和粘贴

### 8.2 半自动提示状态

当设置为 `复制后提示保存到收件箱` 时：

1. 桌面端监听剪贴板变化
2. 如果检测到新的、支持的内容类型
3. 在右下角弹一个轻提示
4. 用户点击 `保存到知栖`
5. 内容被写入 `收件箱`

如果用户不点：

- 提示自动消失
- 不写入任何内容

### 8.3 轻提示形态

本轮只做轻提示，不做复杂弹窗。建议信息包含：

- 标题：`检测到新的剪贴板内容`
- 简短说明：例如 `可保存到收件箱`
- 主按钮：`保存到知栖`
- 次要操作：关闭或忽略

提示层级应高于正文，但不应打断当前窗口主要操作。

## 9. 与正常编辑体验的边界

这是本轮最重要的边界之一。

### 9.1 不替代正常粘贴

用户在知栖中本来就可以：

- 复制一段文字
- 去某个目标页面
- 直接粘贴到光标位置

这条体验完全不动。  
剪贴板捕获不是“新的粘贴系统”，而是“额外的桌面捕获入口”。

### 9.2 知栖前台时不提示

当知栖本身已经在前台，且用户正在知栖内部编辑时，本轮建议不弹捕获提示。

原因：

- 用户可能只是为了在当前页面内复制整理内容
- 如果这时弹“保存到收件箱”，会形成自我打扰
- 这会让剪贴板捕获和编辑器本身的复制粘贴边界混乱

因此本轮规则建议为：

- 只有当知栖不在主要前台编辑态时，才允许半自动提示出现

这个判断可以先做保守，不要求过度精细。

## 10. 去重与过滤

本轮至少要有一层轻量去重。

### 10.1 去重目标

避免这些情况：

- 同一段内容被重复写入剪贴板
- 用户连续按两次复制
- 系统短时间内重复触发相同提示

### 10.2 最小去重策略

保守做法即可：

- 对最近一次已提示或已保存的剪贴板内容生成轻量签名
- 在短时间窗口内，如果内容签名相同，则不重复提示

短时间窗口不需要做成复杂设置项，本轮内部固定即可。

### 10.3 类型过滤

如果内容不在支持范围内：

- 不提示
- 不报错
- 直接忽略

## 11. 桌面端架构设计

### 11.1 前端职责

前端负责：

- 读取系统设置
- 决定是否启用剪贴板捕获
- 展示轻提示
- 用户确认后调用已有 store / repository 链路写入 `收件箱`

### 11.2 桌面层职责

桌面层负责：

- 提供读取当前剪贴板内容的能力
- 在允许时机通知前端“有新的候选内容”

本轮不建议让桌面层直接写数据库。  
仍然保持“桌面层负责感知，前端 store 负责真正落库”的边界。

### 11.3 资源导入复用

对于图片、音频、视频：

- 继续复用已有资产导入与本地资源存储链路
- 生成对应媒体块时使用现有 `assetId / name / mimeType` 结构

这样可以避免重新发明一套媒体保存机制。

## 12. 错误处理与边界

### 12.1 收件箱缺失

如果保存时 `收件箱` 缺失：

- 先按第一阶段既有逻辑自动修复 `收件箱`
- 再写入捕获内容

### 12.2 剪贴板内容无法解析

如果剪贴板格式不稳定、或无法安全识别主内容：

- 不提示
- 不报错打断用户
- 直接忽略

### 12.3 资源写入失败

如果图片、音频、视频导入本地资源失败：

- 不写入残缺块
- 给出一次轻量失败反馈
- 不影响后续普通编辑操作

### 12.4 应用启动早期

如果剪贴板事件发生时，工作区还没 bootstrap 完成：

- 先等待工作区完成初始化
- 再决定是否提示与保存

避免提示出现了，但内容无处可写。

## 13. 测试要求

本轮至少覆盖这些回归：

### 13.1 设置

- 默认值为 `关闭`
- 设置切换后能持久化

### 13.2 捕获候选识别

- 文本可识别
- 图片可识别
- 音频/视频文件可识别
- 不支持类型会被忽略

### 13.3 去重

- 同一内容短时间内不会重复提示

### 13.4 收件箱写入

- 保存文本后，`收件箱` 末尾出现说明块和文本块
- 保存图片/音频/视频后，`收件箱` 末尾出现对应媒体块
- `收件箱` 缺失时会先自动修复再保存

### 13.5 编辑体验边界

- 普通页面里的复制粘贴行为不回归
- 知栖前台编辑态下不弹半自动提示

## 14. 后续阶段接口

本轮完成后，方向 2 可以顺着下面顺序继续长：

1. 全局快捷键拉起快速输入
2. 快速输入默认进入 `收件箱`
3. 托盘增加最近页面 / 搜索工作区
4. 捕获来源信息
5. 附件块与任意文件捕获
6. 再考虑剪贴板历史与更强规则系统

也就是说，这一版先把“复制后可轻确认收入收件箱”做通，而不是把整套桌面收集系统一次做满。

## 15. 结论

这一版的核心判断是：

- “手动捕获”没有必要单独做成系统模式
- 真正值得做的是“关闭 / 半自动提示保存到收件箱”
- `收件箱` 继续复用普通页面模型
- 资源与媒体继续复用现有资产链路
- 内容范围先收敛到文本与现有媒体块能稳定承载的类型

这条路径最轻、最稳，也最适合现在马上进入实现计划。
