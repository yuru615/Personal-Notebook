# 可创建单选/多选交互设计

**日期：** 2026-07-05

**目标：**
为数据表中的单选、多选字段补上“直接输入内容，按 Enter 立即创建选项并选中”的交互，并抽出一个共享模块，同步用于页面属性中的单选、多选编辑。

## 范围

- 调整数据表单元格中的 `select` / `multiSelect` 编辑体验
- 调整页面属性中的 `select` / `multiSelect` 编辑体验
- 抽取共享的可创建选项选择器组件
- 保持现有存储结构、字段值结构、颜色规则不变

## 不在本次范围

- 不改数据表字段管理器里的选项维护界面
- 不新增颜色自定义、排序、拖拽、批量创建等高级能力
- 不新增依赖
- 不重构已有日期选择器方案

## 用户体验目标

### 1. 数据表单选

- 打开下拉层后，顶部直接可输入
- 输入已有选项时，可继续点击已有项完成选择
- 输入新内容后按 `Enter`，立即创建新选项并设为当前值
- 创建或选择完成后关闭弹层

### 2. 数据表多选

- 打开下拉层后，顶部直接可输入
- 输入新内容后按 `Enter`，立即创建新选项并加入当前值
- 选择已有项时继续沿用当前多选切换行为
- 创建或切换后弹层保持打开，方便连续添加

### 3. 页面属性单选

- 从当前“只显示按钮列表”改为与数据表一致的可输入选择器
- 支持输入新内容并按 `Enter` 创建
- 选中或创建后立即保存并关闭

### 4. 页面属性多选

- 从当前“逗号分隔文本输入”改为与数据表一致的可输入选择器
- 支持输入新内容并按 `Enter` 创建
- 支持连续添加多个标签

## 方案对比

### 方案 A：只在数据表里补 Enter 创建

优点：
- 改动最少

缺点：
- 页面属性会继续保留另一套逻辑
- 后续体验和样式会持续分叉

### 方案 B：抽共享可创建选项选择器，两个入口共用

优点：
- 共享输入、过滤、创建、选中逻辑
- 页面属性和数据表体验统一
- 改动集中，后续扩展成本低

缺点：
- 比局部修补多一个共享组件抽取动作

### 方案 C：直接把数据表编辑器整体通用化

优点：
- 复用表面上最多

缺点：
- `CellEditor` 里混着日期、文本、checkbox、浮层定位，不适合直接抽通用
- 容易把页面属性也绑进数据表语义

## 推荐方案

采用方案 B。

只抽“可创建选项选择器”这一层共享能力，不去强行合并数据表和页面属性的全部编辑器。宿主继续各自负责：

- 当前值展示
- 打开/关闭时机
- 新选项如何持久化
- 选中后是否关闭

共享组件只负责：

- 输入框
- 选项列表
- 根据输入显示匹配结果
- `Enter` 创建新选项
- 通知宿主“选中了已有项”或“请求创建新项”

## 组件设计

计划新增共享组件：

- `src/components/shared/CreatableOptionPicker.tsx`

建议职责：

- 接收已有选项列表、当前选中值、单选/多选模式
- 内部维护输入草稿
- 渲染输入框和筛选后的选项列表
- 当输入值与现有标签完全不重复时，展示“创建 xxx”
- 按 `Enter` 时优先触发创建，其次可选中当前高亮项

建议最小接口：

- `mode: 'single' | 'multiple'`
- `options`
- `selectedLabels`
- `placeholder`
- `emptyLabel`
- `onSelect(label)`
- `onDeselect?(label)`
- `onCreate(label)`

这里不把“浮层定位”放进共享组件，继续由宿主处理，避免共享组件被 DOM 定位细节污染。

## 数据与持久化设计

### 数据表

- 仍使用现有 `Property.config.options`
- 仍使用现有 `updatePropertyOptions`
- 仍使用现有 `updateRecordValue`
- 新建选项时：
  1. 生成新 option
  2. 走现有选项归一化逻辑保留颜色规则
  3. 再写入单元格值

### 页面属性

- 仍使用现有 `pageProperties[].config.options`
- 仍使用现有 `setPagePropertyOptions`
- 仍使用现有 `setPagePropertyValue`
- 新建选项时：
  1. 为当前属性追加 option
  2. 保存属性定义
  3. 更新页面属性值

## 颜色策略

- 数据表继续沿用现有 `normalizeSelectOptions` 的颜色分配逻辑
- 页面属性新增选项时，颜色策略尽量与数据表保持一致

本次不单独设计第二套页面属性颜色算法，避免同类选项在不同区域出现不一致的视觉结果。

## 宿主接入设计

### 数据表宿主

修改文件：

- `src/components/dataTable/components/table/CellEditor.tsx`

接入方式：

- 保留现有浮层和定位逻辑
- 把原本的按钮列表替换为共享选择器
- 在宿主里处理：
  - 单选清空
  - 多选增删
  - 创建选项后更新属性 options
  - 创建完成后写入单元格值

### 页面属性宿主

修改文件：

- `src/components/editor/PagePropertiesPanel.tsx`

接入方式：

- `select` 和 `multiSelect` 都切到共享选择器
- 不再使用 `multiSelect` 的逗号文本输入模式
- 可继续用轻量浮层或行内编辑容器承载共享选择器，但交互要与数据表一致

## 风险与处理

### 1. 数据表创建选项需要同时更新字段定义和单元格值

处理：
- 在宿主里把“更新 options”和“更新当前值”串成一个明确顺序
- 保持最小改动，不引入新的全局事务抽象

### 2. 页面属性当前没有现成的共享颜色分配工具

处理：
- 本次优先复用已有规则或抽一个很薄的共享颜色辅助函数
- 不为了这件事重构整套属性 domain

### 3. 多选弹层关闭时机不一致

处理：
- 单选：选中后关闭
- 多选：选中或创建后保持打开

这个规则固定下来，避免后面两个入口行为不同。

## 验收标准

- 数据表单选支持输入并回车创建
- 数据表多选支持输入并回车创建
- 页面属性单选支持输入并回车创建
- 页面属性多选支持输入并回车创建
- 两个入口复用同一个可创建选项选择器模块
- 不新增依赖
- 相关测试通过
