# 思维导图基于白板独立分叉设计

- 日期：2026-06-21
- 项目：`E:\Workspace\P001_Notion个人知识库\notion-web`
- 目标：放弃旧思维导图实现，直接从当前稳定白板代码复制出一套独立实现，作为新的思维导图底座；后续所有思维导图专属能力都只在这套新目录上开发，不再继续修补旧思维导图运行时。

## 1. 结论

本次采用的主方案是：

- 直接复制白板整套能力到新的思维导图目录
- 新目录、类型、路由、块入口、预览、导入导出、存储记录全部独立
- 第一阶段先得到一套“白板同款能力”的稳定思维导图底座
- 第二阶段再在这套底座上做思维导图专属逻辑

本次明确不采用以下方案：

- 不继续修旧思维导图代码
- 不先抽白板与思维导图共享内核
- 不在新思维导图目录里继续 import `src/components/whiteboard/*`

这是一个刻意接受重复代码、优先保证隔离性的设计。当前目标不是最优抽象，而是最小风险重建。

## 2. 产品目标

用户可以在知识库页面中插入一个新的思维导图块，点击后进入独立的思维导图页面。这个页面在第一阶段先具备白板已经验证稳定的交互能力，包括无限画布、缩放、平移、选择、拖拽、框选、基础连线、上一步和下一步、顶部悬浮工具栏、本地保存、预览、导入导出。

后续所有思维导图能力都建立在这套独立底座上继续开发，包括：

- 节点层级
- 自动布局
- 折叠展开
- 拖动改挂载关系
- 思维导图专属快捷键

## 3. 非目标

以下内容不在本次设计落地范围内：

- 继续兼容或保留旧思维导图实现
- 白板和思维导图立即共享底层运行时
- 先做节点树规则再做新壳子
- 先做公共抽象再复制目录

如果后续验证两套实现长期稳定且边界清楚，再考虑抽共享层；这不是当前阶段任务。

## 4. 总体策略

### 4.1 分阶段策略

本次重建分为两个大阶段：

1. **底座复制阶段**
   - 从白板复制出一套完整、独立、可运行的思维导图壳子
   - 打通块入口、路由、存储、预览、导入导出
   - 能稳定打开、保存、关闭、重开

2. **导图专属阶段**
   - 在新壳子上逐步替换成思维导图自己的节点模型和交互规则
   - 不改白板既有实现

### 4.2 隔离原则

- 新思维导图实现允许从白板“复制代码”，但不允许继续“依赖白板内部实现文件”
- 白板后续可以继续独立优化；思维导图后续也可以继续独立优化
- 两边只允许在更高层的通用类型、公共工具或宿主应用框架中并存，不允许一个模式直接嵌套使用另一个模式的运行时代码

## 5. 目录与模块边界

### 5.1 新目录

新思维导图实现重新建立在：

- `src/components/mindmap/`

第一阶段直接从当前白板目录复制并改名，至少包含以下模块：

- `MindmapPage.tsx`
- `MindmapCanvas.tsx`
- `mindmapModel.ts`
- `mindmapPreview.ts`
- `mindmapDocument.ts` 或等价的运行时文档桥文件
- `legacy/` 目录下的独立运行时文件

### 5.2 新目录职责

`src/components/mindmap/` 负责：

- 思维导图页面壳
- 思维导图画布容器
- 思维导图运行时桥接
- 思维导图预览图生成
- 思维导图快照模型
- 思维导图运行时静态资源

### 5.3 不复制到新目录的内容

以下内容不应直接塞进 `src/components/mindmap/`：

- 编辑器块逻辑
- 路由装配
- 搜索逻辑
- Markdown 导入导出逻辑
- 本地仓库和 store 逻辑

这些仍保留在原应用层，但会新增一套思维导图接线。

## 7. 数据模型

### 7.1 新增独立记录类型

重新引入独立的思维导图内容对象：

```ts
interface MindmapRecord {
  id: string
  title: string
  snapshot: MindmapSnapshot | null
  createdAt: string
  updatedAt: string
}
```

### 7.2 第一阶段快照策略

第一阶段的 `MindmapSnapshot` 可以先复制白板快照结构，保证：

- 宿主应用能保存和重开
- 新思维导图页面能复用白板稳定交互
- 第二阶段再逐步替换为节点导向的真实导图快照

也就是说，第一阶段是“思维导图类型名 + 白板快照结构”；第二阶段才演进成“思维导图类型名 + 导图专属快照结构”。

### 7.3 页面块结构

重新引入思维导图块：

```ts
interface MindmapBlock extends BlockBase {
  type: 'mindmap'
  mindmapId: string
}
```

块中只保存 `mindmapId`，不直接内嵌思维导图快照。

### 7.4 工作区快照

工作区快照重新包含：

```ts
interface WorkspaceSnapshot {
  pages: PageRecord[]
  boards: BoardRecord[]
  mindmaps: MindmapRecord[]
  settings: WorkspaceSettings
}
```

其中：

- `boards` 继续服务白板
- `mindmaps` 只服务新思维导图

二者平级，不互相替代。

## 8. 入口链路

### 8.1 编辑器命令菜单

恢复并重建 `mindmap` 命令项：

- 标题：思维导图
- 行为：插入一个新的思维导图入口块，并同时创建一个新的 `MindmapRecord`

### 8.2 页面块卡片

页面中的思维导图块使用独立卡片组件，具备：

- 缩略图
- 标题
- 更新时间
- 点击进入

这张卡片实现必须走新的思维导图预览与路由链路，不复用白板块组件。

### 8.3 路由

思维导图页面路由重新启用：

- `/pages/:pageId/mindmaps/:mindmapId`

其页面内容来自新的 `src/components/mindmap/`，不再指向旧思维导图实现。

### 8.4 搜索与侧边栏

思维导图作为和白板平级的另一类内容对象接入：

- 搜索结果
- 最近内容
- 页面关联关系

要求是：

- 能找到被页面引用的思维导图
- 不把孤立脏数据暴露给用户

## 9. 页面壳与 UI

### 9.1 页面壳目标

第一阶段新思维导图页面只要求和白板页拥有一致的页面外壳体验：

- 进入后是纯画布编辑界面
- 不显示左侧知识库侧栏
- 顶部工具栏悬浮在画布内顶部
- 左上角是返回页面

### 9.2 页面壳边界

本次只统一“页面壳”，不强行把导图内部工具栏改成最终导图形态。第一阶段优先复用白板成熟 UI 结构，先保证稳定和一致。

## 10. 导入导出策略

### 10.1 页面级 JSON

页面级 JSON 备份恢复 `mindmaps` 集合：

- 导出工作区时包含 `mindmaps`
- 导入工作区时恢复 `mindmaps`

### 10.2 Markdown 页面包

思维导图和白板一样，单独挂接一条可逆导出导入链路：

- 导出 Markdown 页面包时写入思维导图资源文件
- 导入时能恢复页面块与对应 `mindmapId`

### 10.3 第一阶段约束

第一阶段允许思维导图导出内容先沿用白板风格的资源文件格式，只要：

- 可以稳定恢复
- 对外显示为思维导图资源
- 后续替换内部快照结构时不破坏导入入口

## 11. 实施顺序

建议实现顺序如下：

1. 复制白板目录到新的 `src/components/mindmap/`
2. 批量改名与重命名内部标识
3. 新增 `MindmapRecord`、`MindmapSnapshot`、`MindmapBlock`
4. 恢复 `mindmaps` 存储与仓库读写
5. 恢复 store 中的思维导图创建、更新、保存流程
6. 恢复编辑器中的 `mindmap` 块与命令菜单入口
7. 恢复思维导图路由页
8. 接回搜索、侧边栏、预览、导入导出
9. 跑构建与聚焦回归测试

这套顺序的重点是：先让“新壳子可运行”，再让“外围链路完整”。

## 12. 风险与处理

### 12.1 风险：复制不彻底

表现：

- 新思维导图目录里仍然 import 白板文件
- 改白板时影响思维导图，或改思维导图时影响白板

处理：

- 把“禁止 import `src/components/whiteboard/*`”作为验收检查项

### 12.2 风险：样式串扰

表现：

- mindmap / whiteboard 使用相同类名，彼此覆盖

处理：

- 第一阶段复制后同步改类名前缀
- 页面壳、卡片、工具栏、运行时样式都切成思维导图自己的命名

### 12.3 风险：外围链路漏接

表现：

- 页面里能插块，但无法搜索、导出、导入或重开

处理：

- 把搜索、侧边栏、Markdown、JSON 备份纳入第一阶段验收范围

## 13. 测试与验证

至少覆盖以下验证：

1. 插入 `mindmap` 块时会同时创建新的 `MindmapRecord`
2. 点击思维导图块会进入 `/pages/:pageId/mindmaps/:mindmapId`
3. 新思维导图页可以保存并重新打开
4. 导出 JSON 后包含 `mindmaps`
5. 导入 JSON 后可恢复 `mindmaps`
6. 导出 Markdown 页面包后包含思维导图资源
7. 导入 Markdown 页面包后可恢复思维导图块
8. 搜索能找到被引用的思维导图
9. 白板现有入口和功能无回归
10. 搜索 `src/components/mindmap/` 时不应引用 `src/components/whiteboard/*`

## 14. 验收标准

完成第一阶段后，应满足：

- 用户可以重新插入思维导图块
- 点击后进入新的思维导图页面
- 页面能像白板一样稳定操作
- 关闭再打开仍然存在
- 可以导入导出
- 白板不受影响
- 代码边界已经完全分开

## 15. 后续版本入口

在第一阶段完成后，第二阶段按以下顺序开发导图专属能力：

1. 节点模型替换
2. 自动布局
3. 节点折叠
4. 节点快捷键
5. 拖拽改挂载关系
6. 导图专属连线样式与锚点规则

只有当第一阶段已经稳定独立后，才开始这一阶段。

## 16. 结论

这次思维导图重做的核心不是“继续修旧代码”，而是“把白板稳定能力复制成一套独立壳子，再在这套壳子上演进”。这条路的优先级是：

1. 隔离
2. 稳定
3. 可继续开发
4. 最后才是抽象复用

在当前项目状态下，这是风险最低、最符合用户目标的方案。
