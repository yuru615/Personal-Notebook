# Direction 2 Phase 2 剪贴板捕获 v1 设计

- 日期：2026-07-08
- 状态：待用户复核
- 适用项目：知栖桌面端
- 主题：方向 2 第二阶段最小剪贴板捕获闭环
- 说明：本稿用于替代和收口 [2026-07-07-direction2-phase2-clipboard-capture-design.md](./2026-07-07-direction2-phase2-clipboard-capture-design.md) 中已经过时的范围与交互口径

## 1. 背景

方向 2 要解决的核心不是“组织得多漂亮”，而是“任何东西都能更快进来，而且全程本地、顺手”。

第一阶段已经把桌面入口地基铺好：

- 固定 `收件箱`
- 左侧 `系统` 分组
- 托盘入口
- 设置中心里的 `剪贴板捕获` 设置项

下一步最自然的补强，不是立刻上全局快捷键、小窗速记或捕获历史，而是先把这条链路做通：

1. 用户在别的应用里复制内容
2. 知栖识别到“这是一份值得收进来的内容”
3. 通过系统右下角通知做一次轻确认
4. 用户确认后，内容直接进入 `收件箱`

本轮的目标不是把知栖做成完整的后台采集中心，而是先把“复制后可轻确认收入收件箱”做成稳定闭环。

## 2. 本轮目标

本轮只做以下事情：

- 复用现有设置项 `关闭 / 复制后提示收进收件箱`
- 桌面端检测新的可捕获剪贴板内容
- 通过系统级右下角通知给出确认动作
- 用户确认后，把内容稳定写入 `收件箱`
- 全程复用现有页面、块、资源库、store、repository 和持久化链路

验收标准不是“已经像完整桌面收集箱”，而是：

- 开启捕获后，能稳定识别支持的剪贴板内容
- 知栖在后台时，仍可发出系统通知
- 用户确认后，内容静默进入 `收件箱`
- 知栖内部正常复制粘贴体验不受干扰

## 3. 本轮不做

本轮明确不做：

- 单独的“手动捕获模式”
- 剪贴板历史列表
- 捕获失败重试中心
- 多条候选队列管理
- 来源应用识别
- PDF / Word / Excel / 压缩包 / 任意文件附件
- 独立的收件箱数据模型
- 收件箱专属卡片 UI
- 全局快捷键拉起输入窗
- 捕获后自动打开主窗口

这里不做“手动捕获模式”，是因为它没有额外产品价值。用户如果想把内容精确放到某个页面，本来就可以正常复制后自己去目标页面粘贴，不需要再额外包一层系统模式。

## 4. 已确认的产品决策

以下边界已经确认：

- 剪贴板捕获设置继续沿用现有两档：
  - `关闭`
  - `复制后提示收进收件箱`
- 不新增单独的 `手动` 档位
- 只要复制动作发生在知栖应用内部，就不弹系统捕获通知
- 同一内容短时间内重复复制时，只提示一次
- 用户点击通知里的确认后，静默收进 `收件箱`，不拉起主窗口
- 第一版支持 `文字 + 图片`
- 图片来源按“能收的都收”处理：
  - 剪贴板里的位图图片
  - 网页或应用里复制出的图片内容
  - 资源管理器里复制的图片文件
- 如果复制的是图片文件，收取时复制一份进知栖资源库
- 文本尽量保留基础格式：
  - 粗体
  - 斜体
  - 下划线
  - 链接
  - 换行
- 复杂样式一律允许降级成普通文本

## 5. 支持内容范围

### 5.1 本轮支持的内容

本轮只支持两大类：

1. 文本
2. 图片

### 5.2 文本范围

文本候选包括：

- 纯文本
- 带基础富文本样式的文字
- 带链接的文字

### 5.3 图片范围

图片候选包括：

- 剪贴板位图
- 从网页或应用中复制出的图片数据
- 资源管理器中复制的图片文件

### 5.4 本轮不支持的内容

以下内容即使出现在剪贴板里，也先忽略：

- 音频
- 视频
- PDF
- Word
- Excel
- 压缩包
- 任意非图片文件
- 同时混合多种复杂格式，且无法稳定判定主内容时

这样做的原因很简单：当前编辑器、资源库和块系统已经对文本与图片有稳定承载能力；如果这一轮把任意文件一起纳入，就会从“接现有地基”升级成“补新的附件体系”，范围会立刻变重。

## 6. 收件箱落地方式

### 6.1 收件箱仍然是普通页面

本轮继续复用既有结论：

- `收件箱` 仍然是一张普通页面
- 不新增 `capture_item`
- 不新增 `clipboard_entry`
- 不新增新的数据表或独立捕获存储

这样可以直接复用：

- 普通页面编辑器
- 现有块渲染
- 现有搜索
- 现有导入导出
- 现有本地持久化

### 6.2 每次捕获写入的结构

每次成功收取时，往 `收件箱` 末尾追加一组普通块：

1. 一行说明块：`剪贴板捕获 · YYYY-MM-DD HH:mm`
2. 实际内容块
3. 一条空白分隔块

不新增“捕获卡片”“捕获容器块”或专属视觉壳。先把内容真实稳定地存下来，比额外包一层样式更重要。

### 6.3 文本如何落块

文本内容按“段落分块、段内保留换行”的规则写入：

- 段落与段落之间：拆成多个段落块
- 同一段里的换行：保留在同一个段落块里

例如：

```text
第一段第一行
第一段第二行

第二段
```

会落成：

1. 说明块
2. 一个段落块：包含“第一段第一行 + 第一段第二行”
3. 一个段落块：包含“第二段”
4. 空白分隔块

这意味着“按段落拆开”指的是 `分成多个块`，而不是把多段内容强塞进一个块里。

### 6.4 富文本格式保留策略

第一版只尽量保留基础格式：

- 粗体
- 斜体
- 下划线
- 链接
- 换行

以下内容统一允许降级：

- 标题层级
- 列表语义
- 复杂嵌套样式
- 颜色、字号、背景等复杂格式

策略是“能稳保就保，保不稳就退回普通文本”，不为了追求样式完整把内容收坏。

### 6.5 图片如何落块

图片统一先进入知栖资源库，再作为普通图片块写入页面：

- 剪贴板位图：生成时间戳文件名后写入资源库
- 剪贴板图片数据：按图片资产导入
- 图片文件：复制一份进资源库，尽量保留原文件名

页面里最终显示的仍然是正常图片块，不额外加捕获壳。

## 7. 候选识别规则

### 7.1 识别顺序

每次检测到剪贴板变化时，按以下优先级识别：

1. 图片文件
2. 剪贴板位图 / 图片数据
3. 富文本 / 纯文本

这样做是为了避免同一份图片文件既被当文件、又被当普通文本路径处理。

### 7.2 无法稳定判定时的处理

如果候选内容主类型无法稳定识别：

- 不提示
- 不报错打断用户
- 直接忽略

本轮原则是“宁可少收，不乱收”。

## 8. 通知交互

### 8.1 触发条件

只有满足以下条件时才允许触发系统通知：

- 设置项为 `复制后提示收进收件箱`
- 当前内容类型在支持范围内
- 当前内容不是短时间内的重复内容
- 当前复制动作不是发生在知栖内部
- 工作区已完成 bootstrap，且可安全写入

### 8.2 通知形态

本轮采用系统级右下角通知，而不是应用内弹窗。

通知建议内容：

- 标题：`检测到新的剪贴板内容`
- 正文：`可收进收件箱`
- 主动作：`收进收件箱`

不要求通知承载复杂预览、二次编辑或多操作分支。

### 8.3 确认后的行为

用户点击 `收进收件箱` 后：

- 不拉起主窗口
- 不自动跳转页面
- 直接静默写入 `收件箱`

这样知栖在后台时也不会打断用户当前工作流。

### 8.4 忽略后的行为

如果用户关闭通知或不点：

- 不写入任何内容
- 不再追着弹第二层应用内确认

## 9. 与应用内复制粘贴的边界

这是本轮最重要的边界之一。

### 9.1 不替代正常粘贴

用户在知栖内部本来就可以：

- 复制一段内容
- 去任意页面
- 直接粘贴到目标位置

这条体验完全不动。剪贴板捕获不是新的粘贴系统，而是额外的桌面捕获入口。

### 9.2 知栖内部复制时不弹通知

当复制动作发生在知栖应用内部时：

- 不弹系统捕获通知
- 不把这次复制当成候选捕获内容

原因是：

- 用户可能只是为了当前页面内整理内容
- 此时弹“收进收件箱”会形成自我打扰
- 会让“普通复制粘贴”和“系统捕获”边界混乱

实现上可以采用保守策略：前端在知栖内部复制时打一个很短的内部复制抑制标记；桌面端即使观察到剪贴板变化，也在这段窗口内忽略。

## 10. 去重规则

### 10.1 去重目标

至少要避免以下情况：

- 用户连按两次复制
- 系统短时间内重复触发同一内容变化
- 一条通知还没处理，又叠出同内容的第二条

### 10.2 最小去重策略

保守做法即可：

- 对最近一次已提示或已处理内容生成轻量签名
- 在固定短时间窗口内，如签名相同，则不再重复提示

这条短时间窗口先作为内部固定规则，不做成额外设置项。

## 11. 桌面端与前端边界

### 11.1 桌面端职责

桌面端负责：

- 读取系统剪贴板候选内容
- 识别支持类型
- 发出系统通知
- 把通知确认动作回传给前端

### 11.2 前端职责

前端负责：

- 读取和判断捕获设置
- 标记知栖内部复制抑制窗口
- 决定是否接受候选
- 把候选转换为现有块结构
- 复用现有 store / repository 链路写入 `收件箱`

### 11.3 不让桌面端直接写库

本轮不建议让桌面端直接写数据库。

仍然保持：

- 桌面端负责“感知与通知”
- 前端 store 负责“最终落库”

这样可以最大限度复用现有业务链路，也减少新旧两套写入逻辑分叉。

## 12. 错误处理与边界

### 12.1 收件箱缺失

如果保存时 `收件箱` 缺失：

- 先按既有逻辑自动修复 `收件箱`
- 再写入捕获内容

### 12.2 文本解析失败

如果富文本解析不稳定：

- 先退回纯文本
- 仍尽量保留段落与换行

### 12.3 图片入库失败

如果图片资源导入失败：

- 不写入残缺图片块
- 给出一条系统级失败通知，例如 `收进收件箱失败`

### 12.4 工作区尚未就绪

如果剪贴板事件发生时工作区尚未 bootstrap 完成：

- 不立即提示
- 等待工作区完成初始化后再决定是否处理

避免通知点下去后无处可写。

## 13. 测试要求

本轮至少覆盖这些回归：

### 13.1 设置

- 默认值为 `off`
- 设置切换后可持久化

### 13.2 候选识别

- 纯文本可识别
- 基础富文本可识别
- 位图图片可识别
- 图片文件可识别
- 不支持类型会被忽略

### 13.3 去重

- 同一内容短时间内只提示一次

### 13.4 应用内边界

- 知栖内部复制时不弹捕获通知

### 13.5 收件箱写入

- 文本保存后，`收件箱` 末尾出现说明块、多段落块和空白分隔块
- 图片保存后，`收件箱` 末尾出现说明块、图片块和空白分隔块
- `收件箱` 缺失时会先自动修复再保存

## 14. 后续阶段接口

本轮完成后，方向 2 可以顺着下面顺序继续长：

1. 全局快捷键拉起快速输入
2. 快速输入默认进入 `收件箱`
3. 托盘增加最近页面 / 搜索工作区
4. 捕获来源信息
5. 任意文件与附件块
6. 候选历史与失败重试

也就是说，这一版先把“复制后可轻确认收入收件箱”做通，而不是把整套桌面收集系统一次做满。

## 15. 结论

这一版的核心判断是：

- 不单独做“手动捕获模式”
- 真正值得做的是“关闭 / 复制后提示收进收件箱”
- `收件箱` 继续复用普通页面模型
- 文本与图片继续复用现有块与资源库链路
- 桌面端负责感知与通知，前端负责最终落库

这条路径最轻、最稳，也最适合马上进入实施计划。
