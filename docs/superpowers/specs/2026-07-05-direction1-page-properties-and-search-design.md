# 方向 1 设计稿：页面属性与搜索升级

- 日期：2026-07-05
- 状态：待评审
- 适用项目：知栖桌面端
- 主题：方向 1：像 Notion 一样更强的组织与检索

## 1. 背景

当前知栖已经具备页面、白板、数据表、思维导图、媒体块和基础全局搜索能力，能够承载较丰富的知识内容。但普通页面仍然主要是“标题 + 正文块”的结构，缺少统一的页面元信息层；搜索也仍以标题、正文和部分结构化内容的文本匹配为主，还不能很好地理解页面的主题、状态、时间和来源。

这会带来几个明显问题：

1. 页面内容越来越多后，普通页面之间缺少可复用的结构化描述。
2. 搜索结果能找到东西，但不够“解释型”，用户不容易快速判断为什么命中。
3. 组织能力更多依赖人工命名和父子层级，缺少跨层级的主题、状态、时间聚合。
4. 后续如果直接上双向链接、引用块、反向链接等网络化能力，底层“页面如何描述自己”和“搜索如何理解页面”还不够稳。

因此，方向 1 的第一阶段不直接追求完整的 Notion 页面数据库，而是先建立一套轻量、可扩展的页面属性与搜索增强底座。

## 2. 总目标

方向 1 的目标，是让知栖的普通页面从“长文本容器”逐步升级为“可结构化描述、可聚合查找、可关系化延展”的知识对象。

第一阶段聚焦两件事：

1. 让普通页面第一次拥有统一的结构化属性能力。
2. 让搜索第一次真正吃到这些结构化信息，而不是只搜标题和正文。

第一阶段完成后，用户应该能明显感受到：

- 页面不再只是文本，而是可以补充标签、状态、日期和文本属性。
- 搜索结果更像“找知识”，而不是“全文检索字符串”。
- 后续做标签聚合、属性筛选、反向链接、关系化检索时，不需要推翻这期的底层设计。

## 3. 范围选择

本次方向文档采用：

- 总路线图 + 第一阶段详细方案
- 第一阶段技术方向：方向 B
- 第一阶段属性类型：
  - 标签（multiSelect）
  - 状态（select）
  - 日期（date）
  - 文本（text）

本期明确不纳入：

- 复选框属性
- 公式属性
- 页面表格化视图
- 双向链接 / 反向链接
- 引用块 / 同步块
- 高级查询语法

之所以不纳入复选框属性，是因为在当前产品里，它的用途大多可被“状态”或“星标”替代，第一阶段收益不高。

## 4. 路线图

### 阶段 1：页面属性 + 搜索升级

建立页面属性底座，让普通页面支持轻量元信息，并让搜索理解这些属性。

交付重点：

- 页面属性定义与页面属性值
- 页面头部轻量属性区
- 搜索支持属性命中
- 搜索结果分组、筛选、命中来源展示、同页多命中片段

### 阶段 2：属性聚合与组织视图

当页面属性稳定后，将“单页属性”提升为“页面集合组织能力”。

候选交付：

- 按标签聚合页面
- 按状态聚合页面
- 按日期聚合页面
- 轻量属性筛选视图
- 最近编辑 / 最近新增 / 最近未完成等回顾视图

### 阶段 3：页面关系化与知识网络

在页面属性和搜索底座稳定后，继续向页面连接和知识网络扩展。

候选交付：

- 双向链接
- 反向链接
- 页面提及
- 页面关系属性
- 引用块 / 同步块
- 更强的搜索与关系联动

这三阶段的顺序是刻意设计的：

1. 先让页面有结构
2. 再让结构能被组织
3. 最后再让页面之间形成网络

## 5. 第一阶段产品定位

第一阶段的定位不是做“完整页面属性生态”，而是做一个轻量可扩展底座。

它解决的是两个最现实的问题：

1. 普通页面如何被结构化描述
2. 用户如何更快、更准地重新找到这些页面

第一阶段做完后，用户应能做到：

- 给页面打标签、设状态、填日期、补充来源或备注
- 按这些属性重新找到页面
- 在搜索里清楚知道结果为什么出现
- 看到同一页面中的多个相关命中片段，而不是只看到一条模糊摘要

## 6. 数据模型设计

### 6.1 总体原则

页面属性采用“借结构，不借存储；借交互，不借整套状态机”的策略。

也就是说：

- 借用数据表属性模型中的概念和部分类型结构
- 不直接把普通页面做成“只有一条记录的数据表”
- 不复用数据表的 `records / views / filters / sort / group` 这整套状态系统

### 6.2 为什么不直接复用数据表整套实现

数据表的主体结构是：

- 表
- 属性定义
- 多条记录
- 每条记录的值
- 视图、排序、筛选、分组

页面的主体结构是：

- 页面树
- 单个页面
- 页面内容块
- 页面元信息

两者虽然都能“挂属性”，但页面不是数据库记录。若硬复用数据表整套模型，页面系统会被迫承担记录、视图、分组等多余复杂度，后续维护成本会显著增加。

### 6.3 借用数据表的内容

建议借用以下思想和结构：

- 属性定义方式：`id / key / name / type / config / timestamps`
- 选项型属性的 `options`
- `select / multiSelect / date / text` 这些基础类型概念
- 相关 UI 交互经验：
  - 新增属性
  - 改名
  - 改类型
  - 配置选项

### 6.4 不借用的数据表能力

第一阶段不借用：

- `title` 属性
- `formula`
- `DatabaseRecord`
- `DatabaseView`
- `sort / filter / group / hiddenPropertyIds / columnWidths`

### 6.5 页面属性建议结构

建议在工作区层引入“页面属性定义”，在页面层引入“页面属性值”。

推荐建模思路如下：

#### 页面属性类型

- `text`
- `select`
- `multiSelect`
- `date`

#### 页面属性定义

字段建议：

- `id`
- `key`
- `name`
- `type`
- `config`
- `createdAt`
- `updatedAt`

其中：

- `select` / `multiSelect` 的 `config` 中包含 `options`
- `text` / `date` 保持轻量配置

#### 页面属性值

每个页面保存当前页面在这些属性上的值，类型随定义而变化：

- `text` -> `string | null`
- `select` -> `string | null`
- `multiSelect` -> `string[]`
- `date` -> `string | null`

### 6.6 定义和值分层的原因

采用“属性定义共享、页面值独立”的设计，目的是：

1. 让同一个属性可以在多个页面间复用，例如统一的“状态”。
2. 让搜索、后续筛选和未来的聚合视图建立在统一属性集合之上。
3. 降低后续扩展到阶段 2 的改造成本。

### 6.7 标签不再走旁路字段

既然第一阶段选择方向 B，标签不建议继续单独存成 `tags: string[]` 这种旁路字段。

建议将“标签”作为正式页面属性的一种：

- `标签` = `multiSelect`
- `状态` = `select`
- `日期` = `date`
- `来源/备注` = `text`

这样可以避免未来出现“双轨存储”：

- 一部分元信息在 `tags`
- 一部分元信息在 `properties`

## 7. 第一阶段交付清单

第一阶段交付拆为两块：页面属性入口与搜索升级。

### 7.1 页面属性入口

交付内容：

- 普通页面支持轻量页面属性
- 用户可在页面中使用以下类型：
  - 标签
  - 状态
  - 日期
  - 文本
- 支持新增属性定义
- 支持页面填写属性值
- 支持属性值编辑
- 支持属性定义复用

阶段目标：

- 页面第一次具备结构化元信息层
- 元信息不再依赖用户手工写进正文

### 7.2 搜索升级

交付内容：

- 搜索支持页面属性命中
- 搜索结果支持基础分组：
  - 页面
  - 白板
  - 数据表
  - 数据表记录
- 搜索结果支持基础筛选：
  - 全部
  - 页面
  - 白板
  - 数据表
  - 标签
  - 状态
- 支持命中来源展示：
  - 标题命中
  - 正文命中
  - 标签命中
  - 状态命中
  - 日期命中
  - 文本属性命中
  - 媒体文件名命中
- 支持同页多命中片段展示

阶段目标：

- 搜索从“搜到了”升级为“解释得清”
- 用户可以更快判断结果是否就是自己要找的页面

## 8. 交互设计

### 8.1 页面属性区位置

页面属性区建议位于：

- 页面标题下方
- 正文上方
- 属于页面头部元信息层的一部分

推荐结构顺序：

1. 面包屑 / 顶部工具栏
2. 封面
3. 图标 + 标题
4. 页面属性区
5. 正文内容

这样做的原因：

- 属性天然属于页面元信息
- 与正文内容边界清晰
- 用户进入页面后，先理解“这页是什么”，再进入正文

### 8.2 页面属性区样式

页面属性区建议采用紧凑、低干扰的列表形态，不做数据库风格的大型管理面板。

推荐表现：

- 默认是紧凑列表
- 以“属性名在左，属性值在右”的方式呈现
- 若页面暂无属性，仅展示轻量入口，例如“添加属性”

属性值示意：

- 标签：`产品 / 搜索 / 规划`
- 状态：`进行中`
- 日期：`2026-07-05`
- 来源：`和用户讨论后整理`

### 8.3 编辑方式

建议保持“点击即编辑”：

- 标签：支持 token 形式的新增、删除、选择
- 状态：下拉选择
- 日期：日期选择器
- 文本：原地输入或轻量浮层输入

交互目标是：

- 比数据库更轻
- 比正文更结构化

### 8.4 搜索界面形态

搜索弹窗建议分三层：

1. 顶部输入框
2. 一行轻量筛选条
3. 结果区

筛选条第一阶段只放最有价值的几个入口：

- 全部
- 页面
- 白板
- 数据表
- 标签
- 状态

结果区建议按分组标题 + 结果列表展示。

### 8.5 搜索结果信息结构

每条结果不只显示标题，还应包含：

- 命中类型
- 命中片段
- 必要时的所属页面或路径

对于页面结果，应当显示命中来源，例如：

- 标题命中
- 标签命中
- 正文命中
- 状态命中

对于同一页面的多次命中，第一阶段就应支持展示多个命中片段，而不是只保留单条摘要。

### 8.6 视觉原则

两条原则：

1. 页面属性区要轻，不喧宾夺主
2. 搜索结果要信息密，但不要花哨

知栖当前仍以阅读和写作为主，属性是辅助层，不应压过正文主体验。

## 9. 第一阶段明确不做

本阶段不交付：

- 完整页面属性管理生态
- 页面属性排序、隐藏、复杂配置面板
- 页面按属性切换为表格视图
- 公式属性
- 页面之间的关系属性
- 双向链接 / 反向链接
- 引用块 / 同步块
- 高级查询语法

这些能力属于阶段 2 或阶段 3 的主题，不在本期落地。

## 10. 实施顺序

第一阶段建议拆成 3 步，而不是一次全部上线。

### 第一步：页面属性底座

内容：

- 页面属性定义模型
- 页面属性值模型
- 页面头部轻量属性区
- 增删改查跑通

### 第二步：接入搜索索引

内容：

- 搜索识别页面属性
- 命中来源展示
- 同页多命中片段

### 第三步：补筛选与结果分组

内容：

- 搜索结果分组
- 按类型筛选
- 按标签 / 状态筛选
- 排序优化

采用三步走的原因：

- 每一步都能单独验证
- 更利于问题定位
- 降低属性模型、搜索逻辑、搜索 UI 同时改动的风险

## 11. 风险与约束

### 风险 1：页面属性和数据表属性耦合过深

如果直接复用数据表整套状态结构，页面系统会被强行拖入数据库复杂度，后续会越来越像“伪数据库记录”。

控制策略：

- 只借属性定义思想和部分类型结构
- 页面侧单独建模

### 风险 2：搜索结果信息量膨胀

命中来源、同页多命中、结果分组、筛选一起加上后，搜索界面很容易变得臃肿。

控制策略：

- 第一阶段 UI 保持紧凑
- 只展示最有价值的信息
- 不提前加入复杂高级查询能力

### 风险 3：共享属性定义引入兼容问题

一旦工作区层引入共享属性定义，就必须同步考虑：

- 桌面端 SQLite 持久化
- 导入导出
- 默认 seed
- 老数据升级

控制策略：

- 设计期就把导入导出和老数据兼容写进实施计划
- 不允许只改前端内存态

## 12. 验证策略

### 12.1 领域层

验证内容：

- 属性定义与属性值读写
- `select / multiSelect / date / text` 的序列化
- 搜索命中排序与命中来源判断

### 12.2 Store / Repository 层

验证内容：

- 页面属性持久化
- 导入导出兼容
- 老数据升级兼容

### 12.3 UI 层

验证内容：

- 页面头部属性区显示与编辑
- 搜索结果分组
- 标签 / 状态筛选
- 同页多命中展示

### 12.4 回归层

验证内容：

- 现有页面编辑不受影响
- 白板 / 数据表 / 导图搜索保持正常
- 桌面端启动、刷新、持久化不丢数据

## 13. 阶段完成标准

只有当以下体验成立时，才算第一阶段达标：

1. 用户可以在普通页面中添加并填写标签、状态、日期、文本属性。
2. 同一工作区内，页面属性定义可以在多个页面中复用。
3. 搜索能识别属性命中，并展示命中来源。
4. 搜索支持基础结果分组与基础筛选。
5. 同一页面中多个命中点能被用户直接看到。
6. 现有页面、白板、数据表、导图搜索体验不回退。

## 14. 结论

方向 1 的第一阶段，不追求一次做成完整 Notion 页面数据库，而是先建立页面属性与搜索增强这套可扩展底座。

这期真正要解决的是：

- 页面如何结构化描述自己
- 搜索如何更准确地理解页面

当这套底座稳定后，后续再逐步进入：

- 属性聚合与组织视图
- 页面关系化与知识网络

这会让知栖沿着“先有结构、再能组织、最后形成网络”的路径稳步演进，而不是一次性堆入过多复杂能力。
