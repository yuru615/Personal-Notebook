# 统一交换归档格式设计

**日期：** 2026-07-14  
**状态：** 已完成设计，待用户审阅  
**范围：** 页面包、全工作区导入导出，以及对历史格式的兼容

## 背景与目标

目前知栖有两类外部文件：

- 当前页面（含子页面）导出为 ZIP 页面包，内部使用 `page-package.json`。
- 全工作区导出为裸 JSON 快照，恢复时会覆盖当前工作区。

这两种格式不一致；更重要的是，裸 JSON 只保存资源引用，不能携带图片、视频和附件文件。另一个已确认的问题是：2026-07-13 改名后，旧页面包的 `zhixi.page-package` 标识不再被接受，而前端只显示泛化的“导入失败”，使用户无法区分旧包不兼容和文件损坏。

本次目标：用一个可携带媒体文件的 ZIP 容器统一新的外部交换文件，同时保留“导入页面”和“恢复工作区”这两种不同且安全的行为。

## 决策

采用统一扩展名 `.zhiqi` 的 ZIP 容器。统一的是容器和基础清单，不统一导入语义。

| 归档种类 | 导入行为 | 是否覆盖现有数据 |
| --- | --- | --- |
| `page-package` | 导入为新的顶层页面及其子树 | 否 |
| `workspace-backup` | 恢复完整工作区 | 是，必须二次确认 |

不把两者合并成一个“智能导入”按钮。两条入口继续清楚地表达风险：页面包是安全追加，工作区备份是恢复/覆盖。

## 新归档结构

每个新的 `.zhiqi` 文件都是一个标准 ZIP，结构固定如下：

```text
manifest.json
payload.json
assets/manifest.json
assets/<relativePath>
```

`manifest.json` 只描述容器和导入语义：

```json
{
  "format": "zhiqi.exchange",
  "formatVersion": 2,
  "kind": "page-package",
  "createdWith": "1.1.1",
  "createdAt": "2026-07-14T00:00:00.000Z"
}
```

- `format` 恒为 `zhiqi.exchange`。
- `formatVersion` 是容器版本；本次为 `2`。
- `kind` 只能是 `page-package` 或 `workspace-backup`。
- `createdWith` 和 `createdAt` 用于排障展示，不参与兼容性判断。

`payload.json` 根据 `kind` 保存业务数据：

- `page-package`：根页面 ID、页面树、白板、数据表、思维导图和同步块组；它等价于现有页面包的业务内容，但不再携带格式标识和资源元数据。
- `workspace-backup`：完整 `WorkspaceSnapshot`，包括页面、页面属性、白板、数据表、思维导图、同步块组和工作区设置。

`assets/manifest.json` 是资源元数据数组；每项资源在 `assets/<relativePath>` 有对应二进制文件。`payload.json` 不重复写资源元数据，避免两处数据不一致。

## 导出行为

### 页面包

“导出当前页面”继续包含当前页面及全部后代、它们引用的白板/数据表/思维导图/同步块组，以及所有被引用的媒体文件。默认文件名改为：

```text
<页面标题>.zhiqi
```

### 工作区备份

“导出全部”改为完整 `.zhiqi` 工作区备份。它包含整个工作区快照和其中所有被引用的资源文件；不再产生新的裸 JSON 备份。

这是一处有意的最小补强：复用现有 Rust ZIP 写入、资源清单、进度和流式文件复制能力，不添加新的压缩库，也不把大媒体文件经前端内存转一次。

## 导入与兼容策略

### 新格式

导入前先读取 `manifest.json`，再根据 `kind` 和当前入口校验：

- 在“导入页面包”中选择 `workspace-backup`：拒绝，不改任何数据，并提示“这是工作区备份，请使用恢复工作区备份”。
- 在“恢复工作区备份”中选择 `page-package`：拒绝，不改任何数据，并提示“这是页面包，请使用导入页面包”。
- 缺少清单、版本过高、缺少 `payload.json`、资源清单不匹配或资源文件缺失：拒绝，并说明文件损坏或版本不受支持。

页面包导入沿用现有 ID 重写规则，因此同一包可以多次导入而不冲突。工作区恢复保留现有“确认后覆盖”语义；为避免资源 ID 与目标工作区冲突，恢复时也会为归档资源建立旧 ID 到本地 ID 的映射，并重写页面、数据表和同步块中已知的资源引用。

所有结构与资源引用必须先完成校验。任何校验、解压、资源写入或数据库提交失败时，工作区数据库保持导入前状态；导入过程中新增但未被提交引用的资源要清理，不能留下可见的半成品。

### 历史格式

新版本继续接受以下旧文件，但不会再导出它们：

| 历史文件 | 接受方式 | 结果 |
| --- | --- | --- |
| 旧 `zhixi.page-package` v1 ZIP | 页面包入口 | 作为页面包导入 |
| 当前 `zhiqi.page-package` v1 ZIP | 页面包入口 | 作为页面包导入 |
| 旧裸 JSON 工作区备份 | 工作区恢复入口 | 按现有覆盖语义恢复 |

旧 JSON 无法凭空补齐当时没有导出的媒体文件；恢复完成后仍可使用页面内容，但其中指向缺失本地资源的块会沿用现有缺失资源恢复提示。新 `.zhiqi` 工作区备份不会有这个限制。

文件选择器兼容范围：页面包入口接受 `.zhiqi` 与旧 `.zip`；工作区恢复入口接受 `.zhiqi` 与旧 `.json`。扩展名只用于便利筛选，实际格式始终由归档内容判定。

## 错误提示与进度

前端不再吞掉所有底层错误。它会把稳定的存储错误转换为用户可操作的提示，至少区分：

- 使用了错误的导入入口；
- 旧版页面包已兼容迁移；
- 文件不是知栖归档或已损坏；
- 归档版本过高；
- 归档缺少资源文件；
- 本地写入或恢复失败。

导入、导出仍使用已有的桌面路径式流式进度；浏览器回退维持字节数组路径。错误信息不得显示本地绝对路径、MCP 令牌或资源内容。

## 实现边界

- 本次不统一白板单独导入导出的 JSON、思维导图文件、Markdown 导入或普通数据文件；它们是编辑器级互操作格式，不是页面/工作区备份。
- 本次不新增跨版本降级器，不支持未来容器版本；遇到高于支持范围的版本应明确拒绝。
- 本次不改变页面包“新增顶层页面”或工作区恢复“覆盖本地内容”的权限边界。
- 不新增依赖；ZIP、JSON、资源复制和 Tauri 文件对话框均复用项目已有能力。

## 验收与测试

Rust 存储测试覆盖：

1. 新页面包在包含子页面、白板、数据表、思维导图、同步块和媒体时可完整导出、导入及重复导入。
2. 新工作区归档在包含媒体时可恢复完整工作区，资源引用有效。
3. 新格式在错误入口、缺少清单、错误类型、缺失资源、非法引用和事务后期失败时拒绝，且数据库状态不变。
4. `zhixi.page-package` v1、`zhiqi.page-package` v1 和旧裸 JSON 仍可导入到对应入口。

前端测试覆盖：

1. 页面包和工作区备份使用正确的默认扩展名与文件筛选器。
2. 工作区恢复仍要求确认；页面导入不触发覆盖确认。
3. 存储错误映射为明确提示，不再统一显示“请检查页面包格式”。

完整改动后运行相关 Rust/前端测试，并运行 `npm test`、`npm run lint`、`npm run build`。涉及 Tauri 文件和 Rust 命令的改动还需执行 Windows Tauri 打包验证。

## 自检

- 无占位符或待定实现项。
- “统一容器”与“保持两种导入语义”没有冲突：两种文件使用相同 ZIP 外壳，但由 `kind` 决定行为。
- 旧 JSON 的媒体局限已明确，且不承诺无法恢复的二进制文件。
- 变更范围限定在页面包、工作区备份、资源复制和相应提示；未扩展到编辑器级单文件格式。
