# 同步块与引用块设计稿

- 日期：2026-07-06
- 状态：待评审
- 适用项目：知栖桌面端
- 主题：阶段 3 - 页面关系化与知识网络中的同步块 / 引用块

## 1. 背景

知栖已经完成了页面属性、搜索增强，以及普通页面里的页面链接 / 页面提及底座。接下来进入“页面关系化与知识网络”的下一步时，最值得优先做的不是图谱，而是让正文内容本身能够在多个位置稳定复用。

这里有两种用户心智需要同时满足：

1. 我想把一组内容放到别处继续共用，并且改一处全部一起变。
2. 我想在别处引用一组内容，但不希望在那里直接改动它。

如果把“引用块”和“同步块”做成两套完全独立的模型，后面会很快出现重复实现：两套创建入口、两套搜索索引、两套导入导出、两套异常修复。因此本期设计选择把“同步块”作为底层核心能力，“引用块”作为同一套共享内容上的只读实例视图。

## 2. 本期目标

本期目标不是做一个完整的 Notion 克隆，而是交付一套可持续扩展的共享内容底座，满足以下能力：

1. 用户可以把一组连续块创建为同步块。
2. 用户可以在其他位置插入该同步块的同步实例，或者插入只读引用实例。
3. 同步的最小单位是“一组连续块”，而不是单个块。
4. 简单块支持在实例中直接编辑并联动更新。
5. 复杂块先允许显示，但在副本里默认跳回主实例编辑。
6. 搜索、导入导出、页面复制、删除和异常恢复都能理解同步块关系，不产生脏数据。

## 3. 本期不做

为了让第一版稳住，本期明确不做以下内容：

- 同步块嵌套同步块
- 引用块 / 同步块嵌套在新的同步组选区里
- 白板、数据表、导图在副本里的原地深编辑
- 同步块权限系统
- 同步块评论、悬浮预览、历史版本面板
- 同步组图谱视图
- 引用块与同步块的独立底层模型
- 按块级权限控制“哪些实例可写”

## 4. 设计原则

### 4.1 同一套共享底座

底层只保留一套“共享块组”模型。同步块与引用块只是实例模式不同：

- `sync`：可参与同步编辑
- `reference`：只读引用

### 4.2 对用户弱化源概念，对系统保留主实例

用户大部分时候看到的是“这是一块同步内容”，而不是“这是谁的源副本”。但系统内部必须保留一个真实主实例，用于：

- 复杂块跳源编辑
- 主实例删除后的迁移
- 异常恢复
- 搜索与定位的兜底逻辑

### 4.3 同步单位是连续块组

同步内容的单位不是单块，而是一组连续块。这样才能覆盖更贴近真实场景的知识复用，例如：

- 一个标题 + 两段说明
- 一个待办段落 + 一个列表
- 一组混合块内容

### 4.4 先稳住容器，不引入整棵嵌套块树

当前页面内容仍然保持线性 `blocks` 数组，不把整个块模型改造成通用嵌套树。第一版通过新增“同步块容器”块类型来承载共享块组，控制改动边界。

## 5. 方案选择

本期采用以下方案：

### 5.1 共享组独立存储

不把共享内容继续存在页面中的某个“源范围”里，也不做“复制块后强同步”的伪共享模型，而是新增独立的共享组记录。

原因：

- 页面调顺序、拆块、改层级时不会直接破坏共享组本体。
- 页面复制、页面包导出、搜索索引都更容易处理。
- 后续扩复杂块支持时，演进路径更清晰。

### 5.2 页面里放实例容器

页面正文里新增 `synced_block` 容器块。页面本身不直接保存共享组内的真实内容，只保存：

- 这个实例属于哪个共享组
- 这个实例自己的身份
- 这个实例是同步模式还是引用模式

## 6. 核心概念

### 6.1 同步块组

同步块组是共享内容本体，包含一组连续块的真实内容。

### 6.2 同步块实例

同步块实例是某个页面中的一个容器块，用来展示某个同步块组的内容。

### 6.3 主实例

每个同步块组内部保留一个 `primaryInstanceId`。它是系统层面的主实例，用于复杂块编辑跳转、主实例迁移和异常恢复。

### 6.4 引用实例

引用块不是另一种独立共享对象，而是 `mode = 'reference'` 的同步块实例。它渲染相同内容，但默认只读。

## 7. 数据模型

### 7.1 新增块类型

`BlockType` 新增：

- `synced_block`

### 7.2 同步块实例结构

页面正文中的 `synced_block` 建议结构如下：

```ts
interface SyncedBlockInstance extends BlockBase {
  type: 'synced_block'
  groupId: string
  instanceId: string
  mode: 'sync' | 'reference'
}
```

说明：

- `groupId` 指向共享组本体
- `instanceId` 唯一标识当前页面里的这个实例
- `mode` 决定当前实例是同步可写还是只读引用

### 7.3 同步块组结构

工作区层新增共享组记录：

```ts
interface SyncedBlockGroupRecord {
  id: string
  blocks: BlockRecord[]
  primaryInstanceId: string
  createdAt: string
  updatedAt: string
}
```

说明：

- `blocks` 保存共享组的真实内容
- 允许内容里出现普通块、代码块、表格块、媒体块等
- `blocks` 中不允许再出现 `synced_block`

### 7.4 工作区快照扩展

`WorkspaceSnapshot` 需要新增：

- `syncedBlockGroups?: SyncedBlockGroupRecord[]`

Rust 侧 `WorkspaceSnapshot`、`BootstrapPayload`、页面包 `PagePackageManifest` 也需要同步扩展，保证以下数据路径完整：

- 启动加载
- 本地备份 / 恢复
- 页面包导入 / 导出

## 8. 持久化设计

### 8.1 SQLite 新增表

建议新增独立表：

- `zhixi_synced_block_groups`

推荐字段：

- `id TEXT PRIMARY KEY`
- `blocks_json TEXT NOT NULL`
- `primary_instance_id TEXT NOT NULL`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`

### 8.2 为什么不用 settings 承载

同步块是正文主能力，不是轻量设置项。如果把共享组挂进 settings：

- 数据边界会混乱
- 导入导出更难校验
- 搜索与孤儿清理不方便处理
- 后续扩展复杂块会更难维护

因此需要单表持久化。

### 8.3 页面仍保存线性块数组

页面 `blocks` 保持现状，只在其中放入 `synced_block` 实例容器，不额外引入通用树结构。

## 9. 创建与插入交互

### 9.1 从现有连续块创建同步块

用户可以先选中连续块组，再通过块菜单执行“创建同步块”。

执行后：

1. 系统创建一个新的 `SyncedBlockGroupRecord`
2. 把原连续块组迁移到该组的 `blocks`
3. 在原位置用一个 `synced_block` 实例替换原连续块组
4. 该实例默认 `mode = 'sync'`
5. 该实例默认成为 `primaryInstanceId`

### 9.2 通过 `/同步块` 插入

斜杠菜单提供：

- `插入同步块`
- `插入引用块`

进入后展示已有同步块组列表，供用户选择插入。

### 9.3 双入口都保留

本期同时保留：

- 从现有内容创建同步块
- 通过 `/同步块` 插入已有同步组

这样符合知栖当前“块更多菜单 + 斜杠菜单”并存的交互习惯。

## 10. 编辑规则

### 10.1 混合模式

本期采用混合编辑模式：

- 简单块允许在实例中直接编辑
- 复杂块先跳主实例编辑

### 10.2 允许直接编辑的块

第一版直接编辑范围：

- 段落
- 标题 1 / 2 / 3
- 待办
- 无序列表
- 有序列表

当这些块出现在同步组中时：

- 在 `sync` 实例里可以直接编辑
- 改动立即回写共享组
- 所有 `sync` / `reference` 实例同步刷新展示

### 10.3 复杂块处理

以下块允许存在于同步组中，但第一版不在副本里做深编辑：

- 代码块
- 普通表格
- 图片
- 视频
- 音频
- 白板入口
- 数据表入口
- 导图入口

规则：

- 在主实例里允许正常编辑
- 在非主实例里点击相关编辑入口时，提示“前往原始位置编辑”
- 对 `reference` 实例始终只读

### 10.4 为什么不禁止复杂块进入同步组

用户已经明确希望底座不要只服务文本类块。如果第一版完全禁止复杂块进入同步组，会把未来升级路径锁死。因此本期允许它们进入共享组，但把深编辑策略收紧到主实例。

## 11. 引用块定义

### 11.1 底层不是第二套模型

引用块不新增独立持久化结构，只使用：

- `synced_block`
- `mode = 'reference'`

### 11.2 交互规则

引用块实例：

- 正文中显示共享组内容
- 默认只读
- 支持点击跳转到主实例
- 内容变更时和同步块一样自动刷新

### 11.3 取消引用与转为同步

第一版不强求提供“引用块直接转同步块”的专门入口，但内部模型支持未来增加：

- 把当前实例从 `reference` 改成 `sync`

## 12. 删除、取消同步与复制规则

### 12.1 删除实例

删除一个同步块 / 引用块实例时：

- 只删除当前实例容器
- 不删除共享组本体
- 其他实例继续正常存在

### 12.2 删除最后一个实例

当某个共享组的最后一个实例被删除时：

- 自动删除该共享组记录
- 不保留孤儿同步组

### 12.3 主实例删除

如果删除的是当前主实例：

- 自动从剩余实例中选择一个新的主实例
- 推荐策略：按创建顺序最早的剩余实例接任

### 12.4 取消同步

当前实例执行“取消同步”时：

1. 系统将当前共享组内容复制为页面里的普通块组
2. 用普通块组替换当前实例容器
3. 当前实例不再关联共享组
4. 其他实例保持原样

### 12.5 页面复制

如果复制的页面中包含同步块 / 引用块实例：

- 复制后的页面应生成新的 `instanceId`
- 但继续指向原来的 `groupId`

也就是说，页面复制默认保留同步关系，而不是深拷贝出新的共享组。

## 13. 搜索与定位联动

### 13.1 搜索索引策略

搜索不能只索引“共享组本体”，而必须按“实例所在页面”展开。

例如同一个共享组出现在页面 A 和页面 B：

- 页面 A 需要有自己的命中结果
- 页面 B 也需要有自己的命中结果

### 13.2 命中来源

搜索新增来源标签：

- `同步块内容`
- `引用块内容`

### 13.3 跳转规则

点击搜索结果时：

- 优先跳到产生该命中的那个页面实例位置
- 不默认跳到主实例

原因：

- 用户看到哪一页命中，通常就想回到哪一页继续看上下文
- 主实例只在编辑与恢复逻辑里承担系统角色

## 14. 导入导出与备份

### 14.1 工作区备份

SQLite 本地备份、JSON 备份 / 恢复都必须带上：

- `syncedBlockGroups`
- 页面中的 `synced_block` 实例

### 14.2 页面包导出

只要导出的页面树中包含某个同步块 / 引用块实例：

- 就必须把该实例引用到的共享组一起导出

如果导出页树中同时包含同一组的多个实例：

- 导出后它们仍作为同一共享组一并打包

### 14.3 页面包导入

页面包导入后：

- 所有导入的共享组生成新的 `groupId`
- 所有导入的实例生成新的 `instanceId`
- 导入包内部原本属于同一组的实例，导入后仍保持互相同步
- 不允许继续引用原工作区的旧 `groupId`

这样可以避免跨工作区悬挂引用。

## 15. 异常恢复与约束

### 15.1 禁止嵌套同步

第一版明确禁止：

- 把同步块实例放进新的同步组选区
- 把引用块实例放进新的同步组选区
- 在同步组 `blocks` 内再出现 `synced_block`

### 15.2 共享组缺失

如果页面里某个 `synced_block` 实例引用的共享组丢失：

- 页面中显示“同步内容不可用”的占位块
- 提供两个动作：
  - 删除当前实例
  - 尝试解除同步并转成本地块（仅当本地还能拿到最近快照时）

### 15.3 实例与主实例不一致

如果 `primaryInstanceId` 指向不存在的实例：

- 系统在加载时自动从剩余实例中重建一个新的主实例

### 15.4 不允许半同步状态

第一版不允许出现这种中间状态：

- 一部分块还留在页面里
- 一部分块已经进了共享组
- 页面实例和共享组同时都认为自己是内容本体

创建同步块必须是原子替换：连续块组整体迁移为共享组，页面原位置只保留容器实例。

## 16. 测试与验证方向

### 16.1 领域层

需要覆盖：

- 同步组与实例的序列化 / 反序列化
- 主实例迁移
- 取消同步后的本地块还原
- 页面复制时 `instanceId` 重建与 `groupId` 复用

### 16.2 Store / Repository

需要覆盖：

- 创建同步块
- 插入同步块 / 引用块
- 简单块直接编辑联动
- 复杂块跳源策略
- 删除实例与删除最后实例
- 导入导出对共享组的重写

### 16.3 搜索

需要覆盖：

- 同组多实例按页面分别命中
- 来源标签区分同步块与引用块
- 点击结果跳回命中实例

### 16.4 回归

需要保证：

- 现有普通块编辑不回退
- 页面复制不破坏同步关系
- 页面包导入导出不产生坏引用
- 启动、保存、恢复和搜索索引更新不丢数据

## 17. 风险与取舍

### 风险 1：多块组让当前线性块编辑器负担变重

取舍：

- 通过 `synced_block` 容器隔离复杂度
- 不把全量块模型改造成嵌套树

### 风险 2：复杂块联动编辑范围过大

取舍：

- 复杂块允许存在于共享组
- 但副本深编辑统一跳主实例

### 风险 3：导入导出关系重写不完整

取舍：

- 共享组单独进入快照和页面包结构
- 导入时统一重写 `groupId / instanceId`

### 风险 4：同步块与引用块双模型重复建设

取舍：

- 底层只保留共享组 + 实例模式
- 不拆第二套“引用块”存储结构

## 18. 第一版验收标准

以下全部满足，视为第一版成立：

1. 用户可以把连续块组创建为同步块。
2. 用户可以通过 `/同步块` 插入已有同步块或引用块。
3. 同步块与引用块都基于同一共享组模型运作。
4. 简单块可在同步实例中直接编辑并同步刷新所有实例。
5. 复杂块在副本中会明确引导前往主实例编辑。
6. 删除单个实例不会误删共享组。
7. 删除最后一个实例会自动清理共享组。
8. 取消同步后，当前实例会被替换为本地普通块组。
9. 搜索可以识别同步块内容 / 引用块内容，并跳到对应实例位置。
10. 页面复制、工作区备份恢复、页面包导入导出后，同步关系保持正确。

## 19. 后续演进

在这套底座稳定之后，后续可以自然扩展：

- 复杂块在副本里的更细粒度编辑
- 引用块转同步块
- 同步块嵌套
- 同步块悬浮预览
- 同步组引用关系面板
- 与页面关系化、反向链接、知识图谱的进一步联动

本期真正要完成的，不是功能数量，而是给知栖建立一套不容易返工的共享内容底座：

- 用户看到的是同步块与引用块
- 系统内部维护的是稳定的共享组与实例关系
- 搜索、导入导出、复制和恢复都围绕这套底座一致运作
