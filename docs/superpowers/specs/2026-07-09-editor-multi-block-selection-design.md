# 编辑器多选块 v1 设计

- 日期：2026-07-09
- 状态：待用户复核
- 适用项目：知栖桌面端
- 主题：正文编辑器中的多选块第一版

## 1. 背景

当前块编辑器已经具备单块拖动、单块删除、空白行插入、引用块/同步块等能力，但还缺少“像桌面文档工具一样一次选中多块再处理”的基础操作。

这次不追求把编辑器做成完整的块级操作台，而是先补最常用、最稳的一段：

1. 鼠标拖拽框选多个块
2. 选中后直接批量删除
3. 选中后从左侧手柄发起整组拖动

这轮目标是把多选块做成一个不打断现有编辑体验的最小闭环，而不是顺手引入一整套新的工具条、快捷键体系或全局 selection 子系统。

## 2. 本轮目标

本轮只做以下内容：

- 支持鼠标拖拽框选多个块
- 支持块级选中态
- 支持 `Delete / Backspace` 批量删除已选块
- 支持从左侧手柄发起已选块整组拖动
- 支持 `Esc` 清空当前选区
- 支持切页后自动清空选区
- 新增一个设置项，控制框选起点范围

验收标准：

- 用户能在正文里稳定框选多个相邻或不相邻可见块
- 选中后按删除键可以一次删掉整组块
- 选中后从左侧手柄拖动，可以整组移动且保持原顺序
- 普通单块编辑、单块拖动、文本输入、空白行体验不被明显破坏

## 3. 本轮不做

本轮明确不做：

- 浮动批量操作工具条
- `Shift` 连选
- `Ctrl / Cmd` 增减选
- 跨页面选区保留
- 多选复制 / 剪切 / 粘贴
- 框选后批量转换块类型
- 跨容器复杂嵌套拖动规则
- 多选状态的全局 store 共享

这些内容先不进第一版，避免把现有编辑器的交互面一下子拉得过宽。

## 4. 已确认的产品决策

以下决策已经确认：

- 多选第一版优先做“鼠标框选”
- 第一版只提供两种批量动作入口：
  - `Delete / Backspace` 批量删除
  - 左侧手柄发起整组拖动
- “选中后浮动小工具条”不进本轮，记入待做文档
- `Shift` 连选、`Ctrl / Cmd` 增减选不进本轮，记入待做文档
- 框选起点支持两种习惯，并做成设置项：
  - 仅允许从左侧块区/空白区开始
  - 允许从正文区域开始
- 第一版默认值采用更稳的模式：仅允许从左侧块区/空白区开始
- 整组拖动只允许从左侧手柄发起，不支持从正文内容区域直接拖整组

## 5. 方案对比与选型

### 方案 A：本地编辑器状态方案

把多选状态、框选矩形、批量删除和批量拖动全部限制在 [`E:\Workspace\个人知识库-桌面端\src\components\editor\BlockEditor.tsx`](E:\Workspace\个人知识库-桌面端\src\components\editor\BlockEditor.tsx) 及相邻样式里。

优点：

- 改动最小
- 不需要重构 store
- 风险集中在正文编辑器
- 最符合当前阶段的最小闭环目标

缺点：

- 后续做浮动工具条或更复杂快捷键时，还要在这套本地状态上继续扩

### 方案 B：全局 store 共享 selection 方案

把多选状态收进 workspace store，让页面头部、侧边工具或其他组件也能感知当前选区。

优点：

- 长期扩展空间更大

缺点：

- 现在明显过重
- 会波及更多模块和测试
- 很容易为了未来假设把这版拖慢

### 方案 C：独立 selection 子系统方案

额外抽出一套块选区控制层，由 `BlockEditor`、`BlockFrame`、拖拽逻辑共同消费。

优点：

- 结构上更整齐

缺点：

- 现在是典型超前建设
- 文件数和心智负担都会明显变大

### 结论

本轮采用 **方案 A：本地编辑器状态方案**。

原则很简单：先让多选块在当前编辑器里成立，再决定有没有必要把它升格成更大的系统。

## 6. 交互设计

### 6.1 进入多选

用户按下鼠标后，只要满足当前设置允许的起点范围，并且拖动距离超过一个很小的阈值，就进入框选态。

框选矩形以编辑器可视区域为边界，只负责命中当前页面中已经渲染出来的 `.editor-row`。

第一版不做“点一下进入块选中，再继续扩选”的混合模式；进入多选的唯一主路径就是拖拽框选。

### 6.2 选中规则

框选过程中，凡是与框选矩形相交的块行都会进入选中态。

第一版按“当前矩形结果就是当前选区”处理，不做按住修饰键后的并集、差集、切换选中。

这样可以避免和后续 `Ctrl / Cmd` 增减选逻辑混在一起。

### 6.3 退出多选

以下情况会清空当前选区：

- 用户按下 `Esc`
- 用户切换页面
- 用户点击普通正文进入单块编辑
- 用户完成批量删除后
- 用户完成整组拖动后

如果用户只是继续在编辑器空白处点一下，不强制保留旧选区，直接回到普通编辑态。

### 6.4 批量删除

当存在多选时：

- `Delete`
- `Backspace`

都执行“删除当前全部已选块”。

删除完成后：

- 如果前面还有可聚焦块，则焦点回到前一个合适块
- 如果前面没有，则焦点回到底部空白行

### 6.5 整组拖动

当存在多选时，只有从“已选块之一的左侧手柄”发起拖动，才进入整组拖动。

如果从未选中的块手柄发起拖动，则沿用当前单块拖动逻辑。

整组拖动要求：

- 保持原有顺序
- 丢到选区内部时不执行移动
- 拖动预览和落点规则尽量复用当前单块拖动样式

## 7. 设置项设计

本轮只新增一个与多选直接相关的设置项：

- 名称：框选起点
- 值：
  - `safe_zone_only`：仅左侧块区/空白区可开始
  - `content_allowed`：正文区域也可开始

为了避免为一个值先发明整套 `editorPreferences` 容器，这个字段直接落在 `WorkspaceSettings` 顶层。

如果后续真的累计出 3 个以上编辑器级偏好，再考虑收拢为单独对象。

设置中心建议放在“编辑与页面默认”分组下，文案可以写成：

- 仅在块左侧或空白处框选
- 允许从正文区域直接框选

## 8. 技术设计

### 8.1 状态边界

多选 UI 状态只存在于 `BlockEditor` 内部，不进全局 store。

预计需要的本地状态：

- 当前是否正在框选
- 框选起点与当前矩形
- 当前选中的 block id 列表
- 当前是否正在执行整组拖动

这样做的原因：

- 这些状态都是短生命周期 UI 状态
- 不需要跨页面保存
- 不需要让白板、数据表、搜索等模块知道它们

### 8.2 数据持久化

本轮唯一需要持久化的是“框选起点设置”。

也就是扩展 `WorkspaceSettings`，增加一个轻量字段，不新增表，不新增独立数据模型。

### 8.3 与现有拖动逻辑的关系

当前 `BlockEditor` 已有单块拖动相关状态：

- `draggingVisualBlockId`
- `dropTarget`
- `onReorderBlock`

第一版多选拖动不重写整套拖拽系统，而是在现有单块拖动流上补一层判断：

- 如果当前有多选，且拖动源来自选区中的某个手柄，则走“整组重排”
- 否则走原有单块重排

为避免把单块 API 硬拗成批量 API，预计会新增一个专用回调，例如：

- `onReorderBlockGroup?(blockIds, overBlockId, position)`

单块逻辑和多块逻辑各走各的，边界更清楚，也更容易回归。

### 8.4 命中计算

框选命中直接基于已渲染的 `.editor-row[data-block-id]` DOM 矩形计算。

第一版不引入虚拟布局索引、不做离屏块命中、不维护额外坐标缓存。

流程：

1. 框选开始时记录起点
2. 指针移动时生成当前矩形
3. 读取当前页面可见 `.editor-row` 的 `getBoundingClientRect()`
4. 根据相交关系得到选中块列表

这是最直接也最稳的做法，成本最低。

### 8.5 焦点与输入保护

为了不干扰正文输入：

- 只有拖动距离超过阈值才进入框选
- 框选开始后阻止正文进入编辑态
- 普通点击仍优先进入原有编辑逻辑

对于 `content_allowed` 模式，这层阈值尤其重要。第一版先保守处理，宁可少触发，不要让正文点按经常误进框选。

## 9. UI 与样式

### 9.1 选中态

选中态落在 `.editor-row` 这一层，不给块内容额外包新卡片。

视觉建议：

- 很浅的底色高亮
- 一条轻边框或内阴影
- 保持现有块间距与手柄逻辑不变

目标是“明显看得出选中了”，但不能把正文变成一排厚重卡片。

### 9.2 框选矩形

框选时显示一个半透明矩形覆盖层：

- 边框颜色沿用当前编辑器蓝色体系
- 填充透明度较低
- 只在框选进行中显示

### 9.3 拖动态

整组拖动时：

- 被选中的所有块保留选中态
- 当前拖动态可以继续借用现有 `editor-row-dragging`
- 落点高亮继续沿用现有 before / after 指示线

这样不用再发明第二套拖动态视觉。

## 10. 边界与异常处理

第一版按以下保守规则处理：

- 如果页面没有块，不进入多选
- 如果只选中 1 个块，删除和拖动仍然成立，不单独分叉
- 如果拖动目标落在当前选区内部，直接忽略这次重排
- 如果批量删除后目标焦点块不存在，回到底部空白行
- 如果页面切换或块列表更新导致部分已选 id 失效，自动过滤掉失效项

引用块、同步块、媒体块、子页面块、白板块、数据表块、导图块，第一版都按“普通块行”参与选中，不单独发明例外规则。

## 11. 测试策略

本轮测试只覆盖高风险主路径：

1. 框选后产生正确选区
2. `Esc` 能清空选区
3. 批量删除会删除全部已选块，并把焦点回到合理位置
4. 从已选块手柄发起拖动时，会整组重排
5. 从未选块手柄发起拖动时，仍然保持单块重排
6. 切页后清空选区
7. 设置项默认值与持久化正确

样式层至少补一条回归，锁住：

- 选中态 class 存在
- 框选覆盖层 class 存在

不为第一版额外写过多视觉快照测试，避免维护成本先膨胀。

## 14. 结论

这版多选块只做一个最小但完整的编辑器闭环：

- 鼠标框选
- 批量删除
- 整组拖动
- 一个起点范围设置项

不顺手做第二套工具条，不顺手做全局 selection store，也不提前做整套桌面级快捷键体系。
