# DOCX 拖放导入页面设计

**日期：** 2026-07-14
**状态：** 已获用户确认

## 目标

用户将现代 Word `.docx` 文件拖入知栖时，可在本地将其转换为可编辑页面：

- 拖入左侧页面树的空白区域：按文件名创建顶层页面。
- 拖入当前页面正文：在当前页面下创建子页面，并由既有 `createPage` 流程同步生成父页面的 `child_page` 块。
- 新页面保留原始 `.docx` 文件作为附件块。

## 决策

复用已安装的 `jszip` 读取 DOCX（ZIP + OOXML），不新增 npm 或 Cargo 依赖，也不使用在线转换服务。解析模块只负责把 `word/document.xml`、图片关系和媒体二进制转换为页面标题、块和待写入的图片；应用层统一写入图片和原始文件资产。顶层页面复用既有 `createPage`，子页面使用新增的原子 store action，同一次保存中创建页面与父页面的 `child_page` 块。

## 内容映射

| DOCX 内容 | 知栖内容 |
| --- | --- |
| Heading 1 / 2 / 3 | 对应标题块 |
| 普通段落 | 段落块 |
| 有序、无序列表 | 对应列表块 |
| 表格 | 普通表格块 |
| 内嵌图片 | 图片资产和图片块 |
| 原始文件 | 页面末尾的文件附件块 |

文件名（去除扩展名）作为页面标题；正文中的第一个标题仍是正文标题，不重命名页面。旧 `.doc`、损坏的 DOCX 与缺失主文档 XML 会显示可处理的导入错误，不创建半成品页面。

## 范围边界

不承诺还原批注、修订记录、浮动文本框、嵌入对象和复杂排版。首个版本只接收 `.docx`；普通文件拖放继续进入收件箱，现有 Markdown、页面包和完整备份入口不变。

## 验证

1. 解析单元测试覆盖标题、段落、列表、表格、图片关系与无效输入。
2. 应用测试覆盖顶层导入、当前页子页面导入、原始文件块和子页面块同步。
3. 侧栏与页面拖放测试确认仅 `.docx` 使用该流程，其他文件保持原有收件箱行为。
4. 完成后运行全量测试、lint 和前端构建；桌面端再用真实 DOCX 手动复核。
