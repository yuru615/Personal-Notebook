# 思维导图特殊块接入设计

- 日期：2026-06-24
- 项目：`E:\Workspace\P001_知栖\zhixi`
- 来源项目：`E:\Workspace\实验软件\mindmap-web`
- 目标：把实验软件里的独立思维导图项目接入知识库，作为一种新的特殊块；保持原思维导图项目的源码、交互逻辑和界面结构不变。

## 1. 范围结论

本次采用以下主方案：

- 采用“路线 2”：知识库负责外层接入壳，思维导图继续作为独立应用运行。
- 页面正文中新增“思维导图”卡片块。
- 点击卡片进入独立思维导图编辑页。
- 编辑页只增加左上角返回按钮，不增加面包屑、知识库标题栏、知识库工具栏或其他外围 UI。
- 思维导图应用本体通过 iframe 运行，内部继续使用原项目现有操作逻辑。
- 知识库负责每张思维导图的独立存储、打开前灌入、编辑后回收。

本次明确不采用以下方案：

- 不把 `mindmap-web` 的内部组件拆进知识库 React 组件树。
- 不修改 `mindmap-web` 源码来适配多实例。
- 不在正文中直接内嵌一个可交互编辑的思维导图画布。
- 不在思维导图编辑页增加除返回按钮外的知识库页面装饰。

## 2. 产品目标

用户可以在知识库页面中通过命令菜单插入一个“思维导图”特殊块。该块在正文中表现为一个卡片入口，显示思维导图标题、更新时间和基础预览占位。点击卡片后进入独立的思维导图编辑页，页面只保留左上角返回按钮，主体区域完整承载原始思维导图应用。用户编辑完成后返回原页面，后续再次点击同一块时，仍然进入该块对应的那一张思维导图，而不是进入一份共享内容。

本次优先保证以下闭环：

1. 插入思维导图块
2. 点击进入独立导图页
3. 在不改原应用逻辑的前提下正常编辑
4. 返回知识库页面
5. 再次打开时仍能回到该块对应的内容

## 3. 非目标

以下能力不在本次实现范围内：

- 思维导图在正文中的原地编辑
- 思维导图真实缩略图导出或卡片截图缓存
- 同时打开两张不同思维导图并并行编辑
- 思维导图协作、评论、提醒、版本历史
- 思维导图与数据表、白板的联动能力
- 对 `mindmap-web` 的功能增强或 UI 改造
- 自动把思维导图当作侧栏子页面单独展示

## 4. 集成原则

### 4.1 原项目不动

`mindmap-web` 是独立 React 应用，当前通过固定的 `localStorage` key 持久化：

- `mindmap-web.document.v1`

本次接入不修改该项目源码，不改它的交互逻辑，不重写其工具栏、画布、快捷键或布局行为。

### 4.2 知识库只做宿主层

知识库负责：

- 特殊块插入入口
- 思维导图记录的创建与保存
- 路由与返回逻辑
- iframe 容器与生命周期
- 打开前把当前块对应数据写入原应用使用的固定存储 key
- 编辑时把原应用写回的最新内容收回到当前块的记录中

### 4.3 单次单图编辑假设

由于原项目固定使用一个 `localStorage` key，本次版本明确采用以下假设：

- 同一时刻默认只编辑一张思维导图
- 如果未来要支持两个窗口同时打开两张不同导图，需要单独改原项目存储模型

这条限制要在设计和代码中显式写明，避免后续误以为已支持多实例并发。

## 5. 数据模型

### 5.1 新增块类型

在知识库现有 `BlockType` 中新增：

- `mindmap`

对应块结构：

```ts
interface MindmapBlock extends BlockBase {
  type: 'mindmap'
  mindmapId: string
}
```

并将其纳入 `BlockRecord` 联合类型。

### 5.2 新增思维导图记录

在知识库工作区快照中新增独立记录集合：

```ts
interface MindmapRecord {
  id: string
  title: string
  snapshot: unknown
  createdAt: string
  updatedAt: string
}
```

说明：

- `snapshot` 直接保存 `mindmap-web` 使用的整份文档对象。
- `title` 作为知识库侧的索引字段保存，用于卡片、路由回退后的入口显示和后续搜索扩展。
- `title` 的真实来源仍然是原思维导图文档中的 `document.title`；知识库保存时同步镜像到 `MindmapRecord.title`。

### 5.3 工作区快照扩展

现有工作区快照从：

```ts
{
  boards: BoardRecord[]
  dataTables: DataTableRecord[]
  pages: PageRecord[]
  settings: WorkspaceSettings
}
```

扩展为：

```ts
{
  boards: BoardRecord[]
  dataTables: DataTableRecord[]
  mindmaps: MindmapRecord[]
  pages: PageRecord[]
  settings: WorkspaceSettings
}
```

### 5.4 迁移规则

兼容旧数据时：

- 若历史快照中没有 `mindmaps` 字段，启动时补为 `[]`
- 原有页面、白板、数据表数据保持不变

## 6. 路由与页面壳

### 6.1 路由

新增思维导图路由：

- `/pages/:pageId/mindmaps/:mindmapId`

其中：

- `pageId` 表示从哪个知识库页面进入
- `mindmapId` 表示打开哪一张思维导图

### 6.2 页面壳规则

思维导图路由下采用最小外围壳：

- 隐藏知识库左侧侧栏
- 不显示面包屑
- 不显示知识库 `PageHeader`
- 不显示页面大标题、封面、图标、导出入口
- 页面左上角只保留一个返回按钮
- 除返回按钮外，页面主体区域由 iframe 全部铺满

### 6.3 返回行为

点击返回按钮后，回到来源知识库页面：

- `/pages/:pageId`

不额外记录二级返回栈，不引入复杂导航状态。

## 7. 原项目接入方式

### 7.1 接入对象

接入的不是 `mindmap-web` 源码，而是它的构建产物：

- `E:\Workspace\实验软件\mindmap-web\dist`

### 7.2 导入方式

知识库项目将消费这份静态产物：

- 把 `dist` 内容复制到知识库项目 `public` 下的固定目录，例如 `public/mindmap-web/`
- 思维导图路由页中的 iframe 直接加载该目录中的 `index.html`

这样做的目的：

- 不依赖实验软件项目的开发服务在线
- 不把独立项目源码混进知识库 React 代码
- 保证知识库单项目启动后即可访问思维导图

### 7.3 同步策略

V1 不做自动同步实验软件构建产物。

约定如下：

- 当实验软件里的思维导图项目更新后，如需同步到知识库，由开发者重新构建并替换知识库里的静态产物目录。
- V1 允许手动同步，不额外引入脚本化流水线。

这是一个刻意保守的方案，优先保证接入稳定，而不是追求跨项目自动编排。

## 8. 存储接管策略

### 8.1 核心问题

原思维导图项目固定使用一个 `localStorage` key：

- `mindmap-web.document.v1`

如果直接原样加载，多张思维导图会共用同一份内容，无法成为知识库里的独立特殊块。

### 8.2 宿主接管流程

知识库宿主在进入某张思维导图时执行以下流程：

1. 读取当前 `mindmapId` 对应的 `MindmapRecord.snapshot`
2. 在 iframe 加载前，把该 snapshot 写入 `localStorage['mindmap-web.document.v1']`
3. 加载原始 `mindmap-web` 页面
4. 原应用按原逻辑读取固定 key 并完成初始化
5. 编辑过程中，原应用继续按原逻辑向固定 key 自动保存
6. 宿主监听该 key 的更新，把最新内容同步回当前 `MindmapRecord`
7. 页面卸载、返回或 iframe 销毁前，再兜底读取一次固定 key 并落库

### 8.3 标题同步规则

每次从固定 key 回收 snapshot 时：

- 从 snapshot 中读取 `title`
- 同步更新 `MindmapRecord.title`
- 更新 `updatedAt`

这样正文中的思维导图卡片无需单独编辑标题，卡片标题自然反映原应用中的最新标题。

### 8.4 写回触发

优先采用以下触发来源：

- 同源 `storage` 事件
- 路由页离开前的兜底读取
- iframe 卸载前的兜底读取

V1 不要求向原应用注入自定义桥接脚本。

## 9. 插入、复制与缺失恢复

### 9.1 命令菜单入口

在知识库 `/` 命令菜单中新增：

- 标题：`思维导图`
- 描述：`插入一个可点击进入的思维导图卡片`

### 9.2 插入行为

用户选择“思维导图”后，系统立即：

1. 创建一条新的 `MindmapRecord`
2. 其 `snapshot` 使用 `mindmap-web` 默认空文档结构
3. 在当前页面插入一个 `mindmap` 块
4. 块中只保存 `mindmapId`

### 9.3 复制行为

当用户复制思维导图块时：

- 创建新的 `MindmapRecord`
- 复制原记录的 `snapshot`
- 新记录标题在原标题后追加“副本”
- 新块指向新记录

不采用“复制块但共用同一张导图”的行为，避免后续误编辑。

### 9.4 删除行为

删除思维导图块时：

- 只删除块
- 不自动删除底层 `MindmapRecord`

这与当前白板、数据表的保守策略保持一致，避免误删内容。

### 9.5 缺失引用恢复

若块引用的 `mindmapId` 找不到对应记录：

- 卡片显示“思维导图不存在”
- 卡片禁用打开
- 提供“重新创建思维导图”恢复入口
- 恢复时使用同一个 `mindmapId` 重建一份新的空导图记录

## 10. 正文卡片表现

### 10.1 卡片形态

思维导图块沿用现有 `CanvasEntryCard` 体系，视觉上与白板、数据表保持一致。

卡片内容包括：

- 类型标签：`思维导图`
- 标题
- 更新时间
- 右侧“打开”动作提示

### 10.2 预览区策略

V1 不做真实导图缩略图。

预览区采用稳定占位方案：

- 显示带有分支结构感的静态占位视图，或简单的空白思维导图占位文案
- 不依赖 iframe 截图
- 不依赖原项目导出图片能力

原因：

- 不改原项目源码是这次第一约束
- 真实缩略图会引入额外截图、导出或桥接逻辑

## 11. 错误处理

### 11.1 静态产物缺失

若知识库中缺少 `mindmap-web` 的静态产物文件：

- 思维导图路由页显示简单错误空态
- 保留左上角返回按钮
- 不让整个知识库页面崩溃

### 11.2 snapshot 非法

若保存的 `snapshot` 结构非法或解析失败：

- 进入编辑页前回退为一份默认空导图文档
- 同时保留该记录，并在下一次成功保存时覆盖为合法内容

### 11.3 打开不存在记录

若访问的 `mindmapId` 不存在：

- 路由页显示空态与返回按钮
- 不自动跳其他页面

## 12. 测试范围

至少覆盖以下测试：

1. 历史快照缺少 `mindmaps` 时可正常启动
2. 命令菜单可插入 `mindmap` 块
3. 插入时会创建新的 `MindmapRecord`
4. 点击卡片会跳转到正确路由
5. 思维导图路由下隐藏侧栏且只显示返回按钮
6. 每张思维导图打开前会把对应 snapshot 写入固定 `localStorage` key
7. 编辑后返回，记录内容能回写到当前 `MindmapRecord`
8. 两个不同思维导图记录不会在顺序打开时串数据
9. 复制思维导图块会生成新的记录而不是共用旧记录
10. 缺失引用时显示异常卡片并支持恢复

## 13. 版本边界

### V1 必做

- `mindmap` 块类型
- `mindmaps` 独立存储
- 命令菜单接入
- 正文卡片入口
- 独立导图路由
- 最小页面壳：只保留返回按钮
- 静态产物 iframe 接入
- 基于固定 `localStorage` key 的宿主接管与回收

### 后续可跟进

- 真实缩略图
- 搜索结果纳入思维导图
- 孤立思维导图清理入口

- 多窗口并发编辑支持

## 14. 结论

本设计的核心不是改造原思维导图项目，而是把它作为一个完整的独立应用嵌入知识库，并由知识库承担“块入口、路由、记录、存储接管”这层宿主职责。这样可以在不改原项目源码和操作逻辑的前提下，把它收编为知识库的一种特殊块，同时把风险控制在知识库接入层，而不是侵入已有思维导图实现。
