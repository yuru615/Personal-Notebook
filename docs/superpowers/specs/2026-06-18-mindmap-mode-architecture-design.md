# 思维导图模式架构设计

- 日期：2026-06-18
- 项目：`E:\Workspace\P001_Notion个人知识库\notion-web`
- 目标：在现有个人知识库总项目中接入一个可持续演进的思维导图块，并为后续优化提供长期决策依据。

## 1. 设计结论

本次采用 `2C` 方案：

- 共享白板内核
- 独立导图模式层
- 文档中提供 `mindmap` 入口块
- 点击入口块进入独立导图编辑页

该方案的核心不是把思维导图做成白板模板，而是把白板作为底层能力提供者，把思维导图作为拥有独立业务规则的内容类型接入知识库。

## 2. 背景与目标

当前总项目已经具备：

- 文档页块编辑体系
- 独立 `whiteboard` 块与白板页面
- 本地存储、导入导出、预览、路由骨架

本次目标不是快速拼接一个“看起来像导图”的页面，而是建立一套后续可持续优化的边界：

1. 导图作为知识库中的第一类内容对象存在
2. 导图和白板共享基础设施，但不混淆业务规则
3. 后续优化时，能明确区分哪些能力做一次即可复用，哪些能力必须只在导图侧演进

## 3. 方案对比

### 3.1 方案 2A：白板模板版

做法：

- 把思维导图直接视为一种白板模板
- 主要复用白板数据结构和编辑方式

优点：

- 接入速度最快

问题：

- 自动布局、节点树、折叠展开等导图能力会持续侵入白板逻辑
- 长期会导致白板与导图边界混乱

### 3.2 方案 2B：轻度混合版

做法：

- 白板内核为主
- 在其上追加部分导图规则

优点：

- 初期成本适中

问题：

- 中长期仍容易把导图规则塞进白板
- 后续维护会越来越依赖约定而不是结构

### 3.3 方案 2C：共享内核 + 独立模式层

做法：

- 白板提供画布、视口、预览、存储骨架等共享基础设施
- 思维导图在其上实现独立的节点树、布局和交互规则

优点：

- 公共能力和专属能力边界最清晰
- 后续演进成本最低
- 最适合接入当前总项目并长期保留

缺点：

- V1 设计需要更克制
- 前期需要先搭清楚分层

结论：

- 采用方案 `2C`

## 4. 架构边界

### 4.1 核心原则

后续优化遵循一句固定规则：

`基础设施共用，业务规则分治。`

即：

- 能抽到底层的能力，进入共享基础设施层
- 带树结构约束的能力，只进入导图模式层
- 带自由画布语义的能力，只保留在白板模式层

### 4.2 内容对象边界

新增独立的 `mindmap` 内容对象：

- 文档页面中新增 `mindmap` 入口块
- 块本身只保存导图引用信息
- 导图正文数据不存入普通文档块

入口块建议结构：

```ts
interface MindmapBlock {
  id: string
  type: 'mindmap'
  mindmapId: string
}
```

### 4.3 页面边界

- 文档页负责展示入口块与组织上下文
- 导图页负责节点编辑、布局和交互
- 导图编辑页与白板页并列，不互相替代

### 4.4 数据边界

思维导图的数据真相应当是“节点树”，不是白板元素数组。

因此：

- 不直接把 `whiteboard snapshot` 当作导图主存储结构
- 不把节点树塞进文档块内容
- 不把导图当作自由摆放白板的一个子场景

## 5. 能力分层

### 5.1 公共能力层

以下能力后续应被视为共享基础设施。优化一次，白板与导图都能受益：

- 画布视口：缩放、平移、居中、适配视图
- 选择系统：单选、多选、框选、命中检测
- 拖拽基础：元素拖动、连线端点拖动、基础吸附
- 渲染基础：画布刷新、局部重绘、性能优化
- 历史机制：撤销、重做、快照回退
- 存储机制：本地保存、迁移、导入导出、备份
- 预览机制：入口块缩略图、封面渲染
- 页面骨架：独立编辑页、返回文档、工具栏容器、标题栏容器
- 通用交互框架：快捷键注册、右键菜单框架、空状态、异常状态

### 5.2 导图专属能力层

以下能力只属于 `mindmap`，不会自动同步到白板：

- 节点树关系：根节点、父子节点、同级节点
- 导图操作流：新增子节点、新增同级节点、连续扩展
- 自动布局：树形布局、左右分支、整理间距、减少交叉
- 分支样式：层级样式、主次分支规则、颜色继承
- 折叠展开：按节点收起子树
- 导图快捷键：例如 `Tab` 新增子节点、`Enter` 新增同级节点
- 导图约束：连线由树关系生成，不允许自由画布式任意连线
- 大纲互转：节点树与文本大纲双向转换
- 导图阅读模式：聚焦节点、逐层展开、浏览视图

### 5.3 白板专属能力层

以下能力继续只属于 `whiteboard`：

- 自由摆放
- 任意连线
- 图形绘制
- 无结构约束编辑
- 白板式组合排版

## 6. 后续优化同步规则

为了避免未来重复优化，后续所有需求先按下列问题归类：

1. 这是画布、存储、渲染、选择、历史、预览类能力吗？
   - 是：优先下沉到共享层
2. 这是节点树、自动布局、导图操作流类能力吗？
   - 是：只放在导图层
3. 这是自由排布、任意连线、图形绘制类能力吗？
   - 是：只放在白板层

据此可得到明确判断：

- 不是“两边都要重做一次”
- 也不是“所有优化都会自动同步”
- 而是“公共能力做一次共享，专属能力按模式分别演进”

## 7. V1 接入范围

V1 目标是把 `mindmap` 作为总项目中的第一类内容对象接进去，形成最小闭环。

### 7.1 V1 要做

- 新增 `mindmap` 入口块
- 文档中显示导图卡片预览
- 点击卡片进入独立导图页
- 独立存储 `MindmapRecord`
- 提供最小节点模型：
  - 根节点
  - 子节点
  - 同级节点
- 支持最基础编辑流：
  - 新增子节点
  - 新增同级节点
  - 编辑节点文本
  - 删除节点
- 支持基础自动布局
- 支持保存、重新打开、预览同步
- 导出备份时纳入总项目数据
- 补齐路由与异常态

### 7.2 V1 不做

- 自由拖拽改布局规则
- 折叠展开
- 复杂主题样式
- 节点备注、标签、图标
- 大纲与导图双向编辑
- 演示模式
- 协作、评论、提醒
- 高级快捷键体系
- 与数据库或页面引用的深度联动

## 8. 数据模型建议

建议为思维导图建立独立数据结构：

```ts
interface MindmapRecord {
  id: string
  title: string
  rootNodeId: string
  nodes: Record<string, MindmapNode>
  viewport: {
    x: number
    y: number
    zoom: number
  }
  createdAt: string
  updatedAt: string
}

interface MindmapNode {
  id: string
  parentId: string | null
  text: string
  order: number
  side?: 'left' | 'right'
  collapsed?: boolean
}
```

说明：

- `rootNodeId` 明确导图根节点
- `nodes` 以记录表方式存储，便于增删改查
- `side`、`collapsed` 可以在 V1 先预留
- `viewport` 与共享画布基础设施保持一致

## 9. 模块拆分建议

建议在总项目中保持以下模块边界：

- `mindmap model`
  - 节点树增删改查
  - 布局计算
  - 数据校验
- `mindmap page`
  - 独立导图编辑页容器
- `mindmap preview`
  - 文档卡片缩略图
- `mindmap block`
  - 文档中的入口块
- `canvas/shared`
  - 视口、选择、渲染、交互框架、历史机制

避免做法：

- 把导图业务规则散落进白板页面组件
- 让白板页面直接识别导图节点树
- 用临时条件分支在同一套白板组件里硬塞两种语义

## 10. 演进路线

### 10.1 阶段 1：接入期

目标：

- 结构正确
- 能存、能开、能回到文档

范围：

- 入口块
- 独立页
- 最小节点树
- 基础布局
- 存储与备份接入

### 10.2 阶段 2：体验期

目标：

- 操作顺手
- 交互一致

范围：

- 节点创建与编辑流优化
- 快捷键
- 同级节点顺序调整
- 分支样式
- 预览质量与性能优化

### 10.3 阶段 3：能力期

目标：

- 成为知识库内部的强内容形态

范围：

- 折叠展开
- 大纲/导图双向编辑
- 页面引用、块引用
- 导图模板
- 演示模式
- 与白板更深层共享渲染与历史能力

## 11. 决策准则

后续每次新增导图需求时，先判断：

### 11.1 适合下沉到共享层的信号

- 白板和导图都能受益
- 与具体业务语义无关
- 更接近视口、渲染、存储、历史、预览、选择机制

### 11.2 必须留在导图层的信号

- 与树结构有关
- 与自动布局有关
- 与导图专属操作流有关
- 与层级视觉规则有关

### 11.3 必须留在白板层的信号

- 与自由排布有关
- 与任意连线有关
- 与图形绘制有关

## 12. 结论

本设计的重点不是尽快做出一个导图页面，而是先在总项目中建立一条不易走偏的演进路径：

- 白板提供共享底座
- 导图维护独立规则
- 文档页负责组织入口
- 导图页负责承载结构化思考内容

采用 `2C` 方案后，后续优化可以稳定遵循同一原则：

- 底层公共能力优化一次，导图和白板共享收益
- 导图特有能力只在导图侧演进
- 白板特有能力继续保持自由画布定位

这份文档应作为后续 `mindmap` 接入、评审和优化时的基线依据。
